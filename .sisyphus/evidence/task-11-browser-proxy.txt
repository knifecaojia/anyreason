### 3. Nginx Proxy Configuration

**File: `docker/nginx/anyreason.conf` (Lines 76-104)**
```nginx
location /api/ {
    set $backend_upstream "http://backend:8000";
    rewrite ^/api/(?!v1/)(.*) /api/v1/$1 break;
    proxy_pass $backend_upstream;
    proxy_http_version 1.1;
    proxy_set_header Authorization $final_auth_header;
    # ... timeout settings
}
```

**WebSocket Proxy (Lines 106-127)**
```nginx
location /api/v1/tasks/ws {
    set $backend_ws_upstream "http://backend:8000";
    proxy_pass $backend_ws_upstream/api/v1/tasks/ws;
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "upgrade";
    # ... long timeouts for WS
}
```

### 4. Deploy Nginx WebSocket Support

**File: `docker-deploy/nginx/anyreason-https.conf` (Lines 212-231)**
```nginx
location /ws/ {
    set $backend_ws_upstream "http://backend:8000";
    proxy_pass $backend_ws_upstream;
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "upgrade";
    proxy_connect_timeout 60s;
    proxy_send_timeout 3600s;
    proxy_read_timeout 3600s;
}
```

### 7. Server-Side Routes Use Localhost Fallback

**Observation**: Next.js API routes (server-side only) have localhost fallbacks:
```typescript
return process.env.INTERNAL_API_BASE_URL || process.env.API_BASE_URL || 
       process.env.NEXT_PUBLIC_API_BASE_URL || "http://localhost:8000";
```

**Analysis**: These are SERVER-SIDE routes that run in Node.js, not in the browser. They:
- Are NOT exposed to the browser
- Use internal network addresses to connect directly to backend
- Should have `INTERNAL_API_BASE_URL` or `API_BASE_URL` env vars set in production

This is the correct pattern - the browser uses relative URLs that go through nginx proxy, while server-side routes use direct backend connections.
