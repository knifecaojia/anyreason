# Task 4: Proxy Negative Evidence

## What Was Searched (and NOT found)

This file documents the absence of problematic patterns, confirming the proxy setup is correct.

### 1. No hardcoded backend origins in frontend

Searched patterns:
- `localhost:8000` - NOT FOUND
- `127.0.0.1:8000` - NOT FOUND  
- `localhost:3000` (as API target) - NOT FOUND
- Any `https?://` absolute URLs in fetch/axios calls - NOT FOUND

Evidence: Grep search across all `.ts` and `.tsx` files in `nextjs-frontend/` returned no matches.

### 2. No WebSocket constructor in browser code

Searched pattern: `new WebSocket(`
- Result: NOT FOUND
- This means the frontend does not directly create WebSocket connections

### 3. No event-source usage in browser

Searched patterns: `EventSource`, `SSE`, `stream`
- Result: NOT FOUND in components
- Note: SSE infrastructure exists in `openapi-client/core/serverSentEvents.gen.ts` but is not actively used

### 4. No fetch() with absolute URLs

Searched pattern: `fetch(`
- Result: All fetch() calls use relative URLs like `/api/...`
- Examples:
  - `fetch("/api/ai/video-models")`
  - `fetch("/api/ai/scenes")`
  - `fetch("/api/tasks?status=queued,running&size=20")`

### 5. No axios with baseURL override

Searched in `openapi-client/client.gen.ts`:
```typescript
export const client = createClient(createConfig<ClientOptions>());
```

- `createConfig()` returns an empty object `{}`
- No `baseURL` property is set
- Axios defaults to same-origin requests when baseURL is undefined

### 6. Backend URLs in API responses are NOT frontend request origins

The task specifically warns about conflating:
- Frontend REQUEST origins (what the browser sends)
- Backend RESPONSE URLs (what the API returns)

This audit confirms:
- Frontend uses relative paths for requests
- Backend URLs in responses are a separate concern (not examined here, but not conflated)

## Conclusion

All checks for problematic patterns came back negative. The frontend correctly:
1. Uses relative API paths exclusively
2. Does not hardcode backend origins
3. Does not create WebSocket connections directly
4. Routes all API traffic through nginx proxy at /api/

The nginx proxy correctly handles:
1. /api/ → backend API
2. /api/v1/tasks/ws → WebSocket with Upgrade headers
3. / → frontend
4. /_next/static/ → static assets with caching
