# 8周快速交付技术路线脑暴（基于《项目功能规划.md》《fixed-goal.md》）

## 范围与目标（已确认）
- **周期**：8 周
- **必交模块**：A 账号/RBAC + 项目生命周期；B Provider/模型管理；C 资产管理；D 基础画布（流程编排/节点视图）；E 分镜/故事板（最小可用）；F 导出（MP4/XML/PR 任一）
- **约束**：轻量化、前后端分离、FastAPI + Postgres + MinIO、前端 Shadcn/ui + Tailwind
- **快速交付原则**：单体可扩展、强复用、少自研、先流程再效果、AI 能力以“接入+可追溯”为主

## 技术路线备选（2-3条）与取舍
1) **轻量单体 + 既有组件优先（推荐）**
   - 后端：FastAPI 单体分层（模块化），SQLAlchemy + Alembic，JWT + RBAC；任务执行用 FastAPI BackgroundTask 或轻量队列（RQ + Redis）。
   - 前端：Next.js 或 Vite + React，Shadcn/ui + Tailwind；画布用 React Flow；状态 Zustand + TanStack Query。
   - 优点：搭建快、学习成本低、迭代速度高；缺点：后期扩展需重构模块边界。

2) **单体 + 明确领域边界（稍重但更稳）**
   - 将 Provider、资产、分镜、导出设成独立模块（仍在同一服务），强制 DTO/Service 层隔离；通过事件表或 Outbox 记录生成与导出流程。
   - 优点：可维护性更好、迁移到微服务成本低；缺点：初期工程量略增。

3) **核心平台 + 外部工作流服务化（集成型）**
   - Provider 管理、图生视频、导出等由外部服务（如 ComfyUI、liblib）承载；平台仅做“编排+资产一致性+审计”。
   - 优点：极快落地 AI 能力、降低实现复杂度；缺点：依赖外部服务稳定性，需做好鉴权与回流。

**推荐**：路线 1 为主、路线 3 为补（先“接口型”接入外部工作流），在 8 周内最稳。若团队中长期维护为重点，可在路线 1 的基础上吸收路线 2 的分层。

## 推荐架构与模块边界（面向快速交付）
- **后端模块**：
  - Auth/RBAC：组织-项目-角色-资源-动作，JWT + 角色权限映射表。
  - 项目生命周期：项目、阶段、里程碑、状态流转与审计。
  - Provider：模型注册、参数模板、密钥管理、调用记录。
  - 资产管理：角色/场景/道具/特效，版本与引用关系。
  - 分镜/故事板：镜头、镜头参数、故事板区域数据。
  - 导出：最小支持 MP4 或 XML 或 PR（选其一）。
- **前端模块**：
  - 工作台：项目列表 + 状态进度；
  - 画布：React Flow 节点图（文本/资产/镜头/导出节点），只做拖拽、连线、属性面板；
  - 资产库：卡片/列表视图 + 版本比对；
  - 分镜/故事板：四区结构的最小可用版本；
  - Provider 中心：配置、测试、用量与调用记录。

## 数据流与集成策略（快交付版本）
- **核心数据流**：项目 → 剧本/资产 → 分镜/故事板 → 导出。
- **AI 接入**：统一 Provider 接口，支持 LLM/图像/视频/音频的“参数化调用”，先打通调用记录、失败重试、产物回流。
- **资产一致性**：资产绑定镜头/故事板节点时强引用版本；输出时生成引用快照。
- **导出策略**：初期只做一种导出；其余通过“导出任务”状态预留接口。

## 错误处理、测试与交付保障
- **错误处理**：统一错误码；关键任务（生成/导出）记录状态与失败原因；支持幂等与有限重试。
- **测试最小集合**：
  - 后端：RBAC 鉴权、Provider 调用与回流、导出任务状态流。
  - 前端：核心流程 E2E（登录→创建项目→建资产→进故事板→导出）。
- **风险控制**：
  - 需求膨胀：坚决以“流程可跑通”为首目标；
  - AI 模型差异：统一适配层与错误兜底；
  - 导出复杂度：优先 MP4 或 XML，PR 放到下阶段。

## 结论（快速交付路线）
- **选型**：FastAPI 单体 + Postgres + MinIO + 轻量队列（可选 Redis）+ React Flow 画布 + Shadcn/ui。
- **方法**：先流程、后效果；先统一 Provider，再做深度编排；以最少导出类型完成闭环。
- **目标**：8 周内达成“可用闭环”，为后续能力扩展预留接口与数据结构。

## 任务清单（安装顺序驱动 AI 辅助开发）
> 目标是按“先基础设施、再数据结构、再服务、再前端、再联调”的顺序推进；所有非本项目代码组件一律容器化（数据库/对象存储/队列），其余已有外部实例直接接入。

### 0. 准备与基线（第 0.5 周）
- 明确 MVP 交付范围（已选 A/B/C/D/E/F）。
- 建立任务看板与里程碑（周拆解）。
- 统一 API 命名风格与错误码规范（避免返工）。

### 1. 容器化基础设施先行（第 1 周）
- **PostgreSQL**：容器化 + 初始化脚本（建库/用户/权限）。
- **MinIO**：容器化 + Bucket 初始化策略与访问密钥。
- **队列/任务**：容器化 Redis（如需异步任务，配 RQ/Celery）。
- 统一 `.env` 与连接配置，形成“可一键启动”的本地开发基线。

### 2. 数据结构设计优先（第 1-2 周）
- **RBAC 数据结构**：组织/项目/角色/资源/动作/权限绑定与继承。
- **项目生命周期**：项目、阶段、里程碑、状态流转。
- **Provider 模型管理**：Provider、Model、配置模板、密钥引用、调用记录。
- **资产管理**：资产主表、资产版本、引用关系（镜头/故事板）。
- **分镜/故事板**：镜头表、镜头参数、故事板区域布局数据。
- **导出任务**：任务表、状态流、产物记录。
- 使用 AI 辅助产出：ER 图草案 → 评审 → SQLAlchemy 模型。

### 3. 存储设计（媒体资产与数据库配合）（第 2-3 周）
- **对象存储设计**：\n  - 统一路径规范：`project/{id}/asset/{type}/{asset_id}/v{n}/...`。\n  - 产物与源文件分层（raw/derived/preview）。\n  - 文件元信息（尺寸/格式/时长/帧率）回写 DB。\n- **数据库与存储关联**：\n  - DB 记录持有对象存储 key 与版本号；\n  - 镜头/故事板强引用资产版本（不可变快照）；\n  - 删除策略：软删除 + 延迟清理对象存储。\n- 用 AI 生成“媒体元信息模型”与“引用完整性规则”清单，减少漏项。

### 4. 后端服务初构（第 3-5 周）
- FastAPI 项目骨架 + Swagger；基础健康检查与鉴权中间件。
- RBAC 校验 + JWT 登录；最小可用权限校验闭环。
- Provider 管理 CRUD + 调用记录；统一错误返回。
- 资产管理 CRUD + 版本上传/查询。
- 分镜/故事板最小接口（创建/更新/读取）。
- 导出任务最小接口（提交/状态/结果）。
- AI 辅助：从数据结构自动生成 Pydantic Schema 与 CRUD 接口草稿。

### 5. 前端最小设计与实现（第 5-6 周）
- 登录/权限入口 + 项目列表/状态展示。
- Provider 管理页：配置表单 + 测试调用入口。
- 资产库：卡片视图 + 版本切换。
- 画布：React Flow 节点拖拽、连线、属性面板。
- 分镜/故事板：四区最小版（创意区/时间线/预览/素材）。
- AI 辅助：根据接口文档生成前端表单与列表结构草案。

### 6. 前后端联调与联测（第 6-7 周）
- 核心闭环：登录 → 创建项目 → 建资产 → 建镜头 → 画布关联 → 导出任务。
- 权限验证：不同角色访问控制、接口权限校验一致性。
- 统一错误提示与空态。

### 7. UI 与体验验证（第 7 周）
- 观察关键路径是否直觉：入口、导航、画布交互、资产选择、导出流程。
- 根据反馈精简 UI，确保“能用、清楚、少步骤”。\n- AI 辅助：收集问题清单 → 生成改版建议 → 人工确认后落地。

### 8. 后续任务逐步分解（第 7-8 周）
- 导出多格式扩展（若首版只做 MP4/XML/PR 之一）。\n- 任务/审核流补全。\n- 成本/用量统计（如需）。\n- 外部工作流接入（ComfyUI/liblib）与结果回流。

## 依赖与并行建议（快速交付）
- 容器化基础设施与数据结构设计先行，避免后期大范围迁移。
- 存储设计必须早于资产/导出实现，否则会返工。
- 后端与前端可在接口定义稳定后并行推进。
