# Agent 内置工具输入说明（用于日志分析）

本文档总结“Agent 可调用的内置工具（tools）”在本仓库中的定义位置与**每个工具的输入（入参）**，用于你在查看 task work 日志 / trace_events 时快速对照。

覆盖范围：
- Chatbox 场景（`scene_engine/scenes/chat.py`）挂载的 Chatbox 工具：`CHATBOX_TOOL_FUNCTIONS`
- AI 场景测试（`ai_scene_test`）按 `tool_ids` 选择挂载的预览工具：`TOOL_REGISTRY`

## 1. 工具调用机制（pydantic_ai）

- 工具是 Python async 函数，第一参为 `RunContext[Deps]`，后续参数由模型以 JSON 形式传入。
- **工具名（tool-call name）默认等于函数名**（例如 `extract_characters`）。
- 入参 schema 由函数签名 + Pydantic 模型注解推导（例如 `assets: list[AssetDocV1]`）。
- 额外“隐式输入”来自 `ctx.deps`（例如 `ctx.deps.user_id`、`ctx.deps.script_text`）。

## 5. 业务数据存储与 AI 工具如何配合（DB / VFS / ApplyPlan / Markdown）

这一节不是“工具输入对照表”，而是把你关心的三件事一次说清楚：
1) 剧集/分镜/资产等业务数据到底存哪（DB vs VFS），各自该承担什么责任；  
2) AI 工具（Chatbox / 场景测试 / 异步任务）输出如何从“预览”走到“可复用的持久化产物”；  
3) 后期以 Markdown 作为 AI 能力预览与存储的统一格式，如何在不打架的情况下与 DB 协同。

### 5.1 先给结论：推荐的分层与“单一事实来源（SSOT）”

把系统想象成三层：

- **DB（查询索引层）**：存“可查询、可关联、需要事务一致性”的结构化字段（id、状态、编号、关联关系、统计值、索引字段）。
- **VFS（文档/产物层）**：存“人类可读、可导出、可版本化”的文本/文件产物（Markdown、JSON、图片等），以及 AI 运行产生的可回放材料。
- **AI 工具（产出计划层）**：工具不直接写库，而是先产出 `ApplyPlan`（以及 trace），由统一入口决定是否落地为 VFS 产物/是否同步回 DB。

本项目目前已经部分体现了这个结构，但还存在“同一份内容同时存在 DB 和 VFS、缺少显式指针”的情况（见 5.4）。

如果你后期明确要“以 Markdown 作为预览/存储主格式”，建议采用下面这个 SSOT 规则（也是最不容易被 AI 不稳定输出坑到的方案）：

- **剧本文本**：SSOT 建议在“文本文件”（VFS 或 MinIO 对象），DB 仅存摘要/统计/状态/指针与索引字段。
- **分集（EPxxx）人类可读稿**：SSOT 在 VFS 的 Markdown；DB 存 `episode_code/episode_number/title/status/word_count/line_range` 等索引字段，并保存“该集正文 md 的 file_node_id”（当前还没有该字段，只能弱关联）。
- **分镜/故事板**：SSOT 建议在 VFS 的 Markdown（每镜头或每段 storyboard 一个文件）；DB 的 `storyboards` 表作为“可查询投影/缓存”，从 Markdown 解析而来（当前 DB 是主路径，VFS 是产物路径）。
- **资产**：SSOT 分两类：  
  - “索引与生命周期”：DB（`assets/asset_variants` 等）  
  - “可交换的文档描述”：VFS（JSON/Markdown），必要时再同步回 DB

#### 5.2.3 AI 工具输出与落地：ApplyPlan 的真实语义

`ApplyPlan` 本质是一个“可落地的写入计划”，只包含：
- `kind`（目前仅：`episode_save / asset_create / asset_bind`）
- `tool_id`（用于区分来源工具/执行分支）
- `inputs`（真正要写入的内容）
- `preview`（面向 UI 的预览信息）

定义见 [apply_plan.py](file:///f:/animate-serial/apps/anyreason/fastapi_backend/app/ai_tools/apply_plan.py)。

重要现象（决定了你后续怎么统一 Markdown 存储）：
- **Chatbox 工具的 `dry_run` 只是“返回计划”**，工具本身并不执行写入；真正写 VFS 的是统一执行接口：  
  - 计划生成：见 [chatbox_tools.py](file:///f:/animate-serial/apps/anyreason/fastapi_backend/app/ai_tools/chatbox_tools.py)  
  - 计划执行（写 VFS + 写 `file_nodes`）：见 [apply_plans.py](file:///f:/animate-serial/apps/anyreason/fastapi_backend/app/api/v1/apply_plans.py)
- **AI 场景测试的工具也产出 ApplyPlan，但 tool_id 不是可执行的同名分支**：  
  - 场景测试工具 `preview_script_split` 会生成 `ApplyPlan(kind="episode_save", tool_id="preview_script_split", inputs={"episodes":[...]})`  
    见 [tools.py:L118-L191](file:///f:/animate-serial/apps/anyreason/fastapi_backend/app/ai_scene_test/tools.py#L118-L191)
  - `/apply-plans/execute` 当前只接受 `tool_id == episode_save/asset_create/asset_bind` 三种“正式落地分支”，不会执行 `preview_*` 的 tool_id：  
    见 [apply_plans.py:L115-L221](file:///f:/animate-serial/apps/anyreason/fastapi_backend/app/api/v1/apply_plans.py#L115-L221)

因此：**场景测试的 ApplyPlan 目前定位就是“预览”，不是“可直接落库的计划”**；要落库必须做一次“计划归一化”（见 5.5 的建议）。

### 5.3 “Markdown 作为统一预览/存储格式”应当放在哪一层

你后期希望：AI 场景测试、各种需要调用 AI 的地方，都能以 Markdown 预览、并最终落到可复用的 Markdown 存储。结合现有代码，最稳的落点是：

- **Markdown 的归宿：VFS（file_nodes + MinIO）**  
  这是当前唯一对“文本文件”有统一 CRUD、统一鉴权、统一预览入口的存储层。写入方式已经标准化（`create_text_file`）：见 [vfs_service.py](file:///f:/animate-serial/apps/anyreason/fastapi_backend/app/services/storage/vfs_service.py)。

- **DB 的角色：索引与关系**  
  DB 应该保存：某个项目/某一集/某个分镜，当前“正文/产物”的节点 id、状态、统计值，以及用于查询的字段（如 `episode_number`、`scene_number`、`shot_code`、`asset_id` 等）。  
  这能让你做到：DB 用来“找”，VFS 用来“看与改”，AI 用来“生成候选并落档”。

- **AI 工具的角色：产出“可落地的文档+元数据”**  
  最理想的 AI 输出不是“一段长文本”，而是：
  - 面向人：Markdown（可预览、可直接作为文档 SSOT）
  - 面向机：结构化 inputs（ApplyPlan.inputs / JSON）
  两者用同一个 run_id 或同一个文件路径建立绑定。

### 5.4 当前的“协同断点”（你现在觉得混乱，核心就卡在这些点）

下面这些断点并不是“设计错误”，更像是迭代过程里自然出现的“拼图没扣上”，但它们会直接影响你后续“一次性搞清楚并统一到 Markdown”：

1) **分集 Markdown（VFS）与 `episodes` 行缺少显式指针**  
   - `/apply-plans/execute` 能写 `分集/EPxxx.md`，但不会更新 `episodes` 表；  
   - `episodes` 表也没有“正文 md 的 node_id”字段，只能靠文件名弱关联。  
   代码入口见 [apply_plans.py:L115-L149](file:///f:/animate-serial/apps/anyreason/fastapi_backend/app/api/v1/apply_plans.py#L115-L149) 与 [models.py:L474-L516](file:///f:/animate-serial/apps/anyreason/fastapi_backend/app/models.py#L474-L516)。

2) **场景测试的 ApplyPlan 目前不可直接执行**  
   - tool_id 以 `preview_*` 标识来源，执行端只接受“正式 tool_id”。  
   这会导致“预览里的 Markdown 产物”无法一键落库，除非做归一化或扩展执行端。

3) **故事板存在两套“来源”**  
   - DB：`ScriptStructureService.structure_script` 会从剧本文本解析并重建 `episodes + storyboards`  
     见 [script_structure_service.py:L266-L336](file:///f:/animate-serial/apps/anyreason/fastapi_backend/app/services/script_structure_service.py#L266-L336)  
   - VFS：`episode_storyboard_agent_apply` 会生成并写入 `故事板/*.md`（人类可读、可导出）  
     见 [episode_storyboard_agent_apply.py](file:///f:/animate-serial/apps/anyreason/fastapi_backend/app/tasks/handlers/episode_storyboard_agent_apply.py)  
   目前二者没有强制同步关系：你可能出现“DB storyboard 更新了，但 VFS storyboard 还是旧的”或反过来。

### 5.5 推荐的协调方式（不改代码的“统一心智模型 + 未来演进方向”）

这一节的建议分成两类：  
A) 你现在就能用来分析日志/组织产物的心智模型；  
B) 你后期要落地“Markdown 统一预览/存储”时最小改动的演进方向（这里只写思路，不动代码）。

#### 5.5.1 统一心智模型：所有 AI 能力都走“预览 → 可执行计划 → 产物归档”

建议把所有 AI 能力的结果都归一成同一种三段式流程：

1) **Preview（预览）**：产生结构化结果与 Markdown（但不写 DB/VFS）  
   - 场景测试：天然就是 Preview（tool_id 以 `preview_*` 开头，且有 trace）  
     见 [runner.py](file:///f:/animate-serial/apps/anyreason/fastapi_backend/app/ai_scene_test/runner.py)
   - Chatbox 工具：`dry_run=True` 时也是 Preview（返回 ApplyPlan.preview + inputs）

2) **Plan Normalize（计划归一化）**：把不同来源的 plan 变成“执行端认识的 plan”  
   - 例如把 `tool_id="preview_script_split"` 归一为 `tool_id="episode_save"`，保持 `kind` 与 `inputs` 结构不变  
   - 同理：把 `preview_extract_character` 归一为 `asset_create`

3) **Apply（执行落地）**：由一个统一入口写入 VFS，并在必要时同步 DB 索引字段/指针  
   - 当前统一入口已经存在：`POST /apply-plans/execute`（只负责写 VFS）  
   - 后续演进可选：执行后回写 DB 指针（例如 episode 正文 md 的 node_id）

这能带来的最大收益是：你在看日志/trace 时，只要抓住“有没有形成 ApplyPlan.inputs（可执行）”，就知道这次 AI 运行有没有产出“可持久化成果”。

#### 5.5.2 Markdown 统一存储的文件组织建议（面向未来，但不要求现在改代码）

在现有 `分集/资产/绑定/故事板` 之外，建议为“AI 运行归档”单独设一个顶层目录（例如 `AI` 或 `运行记录`），每一次 AI 运行都至少生成两类文件：

- `run.md`：人类可读的运行摘要（Markdown，前端可直接预览）
  - 建议包含：目标、输入摘要（脚本长度/哈希/片段）、选择的 agent/version/model、关键结论、下一步建议
  - 建议附上：本次生成的产物链接（VFS node_id 或路径）与 ApplyPlan 摘要

- `plan.json`：可执行计划（结构化、可回放）
  - 存 `ApplyPlan` 的原始对象（包含 inputs/preview/tool_id/kind/id）
  - 若存在“归一化前/归一化后”的差异，可各存一份，便于审计

这样你就能实现：同一份 AI 运行，既能让人一眼看懂（run.md），又能让系统随时再执行或复现（plan.json）。

#### 5.5.3 DB 与 Markdown 的同步边界（避免以后越改越乱）

建议明确一个原则：**DB 不负责存“长正文”，只存“可查询字段 + 指针”**。结合现有表结构，未来可以按下面边界演进：

- `episodes.script_full_text`：适合逐步转为“导入用缓存/兜底”，而把正式“分集文档”放到 VFS；DB 增加一个 `episode_doc_node_id`（指向 `分集/EPxxx.md`）会极大提升一致性与可追踪性。
- `storyboards`：作为可查询投影；当 VFS 中的 `故事板/*.md` 更新时，通过解析同步 DB；当 DB storyboard 被结构化生成时，也同步生成/覆盖 `故事板/*.md`。
- `assets`：DB 管生命周期与引用关系；VFS 存“资产说明书”（JSON/Markdown），并允许人工编辑后回写 DB（通过解析或工具）。

### 5.6 你在做“日志分析 + Markdown 归档”时，建议优先抓哪些信号

如果目标是“从日志里还原：这次 AI 到底改了哪些业务数据/产物”，建议按优先级抓下面信号：

1) **有没有产生 ApplyPlan.inputs**  
   - 有 inputs：意味着“至少能落到 VFS 文件”（前提是 tool_id 可执行或可归一化）  
   - 没有 inputs：大概率只是对话/分析，没有形成可持久化产物

2) **有没有写 VFS（file_nodes）**  
   - ApplyPlan 执行会写 `file_nodes` 并 put 到 MinIO（对象 key 在 `vfs/...`）

   - 任务 handler 直接写 VFS 的情况，要看 task result 或日志事件（例如故事板 apply 会回写 `storyboard_root_node_id`）

3) **有没有同步回 DB（episodes/storyboards/assets）**  
   - 结构化解析（`structure_script`）会重建 `episodes + storyboards`  
     见 [script_structure_service.py:L266-L336](file:///f:/animate-serial/apps/anyreason/fastapi_backend/app/services/script_structure_service.py#L266-L336)

     见 [episode_storyboard_agent_apply.py:L48-L66](file:///f:/animate-serial/apps/anyreason/fastapi_backend/app/tasks/handlers/episode_storyboard_agent_apply.py#L48-L66)
