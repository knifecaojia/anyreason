## 1. 现状回顾与问题诊断

### 1.1 当前状态

创作工坊目前已实现的能力：

- 基于 ReactFlow 的无限画布，点状网格背景
- 10 种节点类型，分为 4 组（创作、AI 生成、展示、引用）
- 左侧悬浮菜单 + 右侧可折叠 Tab Pages（故事板/资产库/详情）
- VFS 持久化（画布 JSON 存储在 `/创作工坊/画布/` 目录下）
- 批量执行队列（BatchQueueManager，前端拓扑排序 + 并发控制）
- 一键生成工作流（从分集数据自动创建 Script → Slicer → Storyboard → Generator → Preview 链路）
- 节点间数据流传播（Kahn 拓扑排序 + 端口类型匹配）
- 撤销/重做、复制/粘贴、对齐辅助线
- 导入/导出画布 JSON

### 1.2 核心问题

经过对代码和设计文档的全面审查，当前创作工坊存在以下结构性问题：

**问题一：设计哲学缺失，借鉴过度。** 当前 UI/UE 大量借鉴 Tapnow 和开源项目的布局模式，但 Tapnow 是通用图片/视频生成工具，而本项目的核心对象是「故事板」——一个有明确叙事结构、镜头语言、资产绑定关系的领域对象。通用画布工具的交互范式无法直接映射到故事板创作的工作流。

**问题二：节点语义混乱。** 10 种节点类型中存在职责重叠和语义模糊：

- AssetNode（资产节点）和 ReferenceNode（引用节点）都在「引用组」，但用途不同却缺乏清晰区分
- MediaNode（展示组）和 GeneratorNode（AI 生成组）都涉及图片/视频，但一个是静态展示，一个是 AI 生成，用户难以理解何时用哪个
- PreviewNode 的存在意义不明确——GeneratorNode 自身已有结果预览能力

**问题三：数据模型缺乏后端支撑。** 画布数据完全以 JSON 文件形式存储在 VFS 中，没有独立的数据库模型。这意味着：

- 无法对画布/流程进行结构化查询
- 无法建立画布与故事板、资产、任务之间的数据库级关联
- 无法支持定时运行、批量运行等需要后端调度的场景

**问题四：画布与故事板的关系不明确。** 设计文档声称「画布不与剧本/剧集强绑定」，但实际上创作工坊的核心价值就是服务于故事板创作。这种「弱关联」设计导致用户进入画布后不知道自己在为哪个故事板工作，产出的媒体文件也无法自动归属到正确的故事板。

## 2. 全页界面设计分析

### 2.1 当前布局的问题

当前创作工坊嵌套在 `(aistudio)` 路由组的通用布局中，画布区域被限制在 `h-[calc(100vh-8rem)]` 的容器内，顶部有全局导航栏占据空间。对于一个以画布为核心的创作工具，这种布局浪费了宝贵的屏幕空间。

### 2.2 全页界面方案

创作工坊应该采用全屏沉浸式布局，具体策略：

**入口方式：** 用户从任意页面（如故事板列表、分集详情）点击「进入创作工坊」后，应以全页覆盖的方式打开，类似于 Figma 的编辑器模式。不是新开浏览器标签页，而是在当前窗口内以全屏层（z-index 覆盖）的方式呈现。

**布局结构：**

- 画布占据整个浏览器窗口（100vw × 100vh）
- 顶部极简工具栏（高度 40px 左右），包含：返回按钮、画布名称、保存状态、用户头像
- 左侧工具面板（可折叠）
- 右侧数据面板（可折叠）
- 没有全局导航栏、没有侧边菜单栏

**退出方式：** 顶部左上角的返回按钮，或 ESC 键，返回到进入前的页面。

**路由设计：** 建议使用 `/studio/[canvasId]` 作为路由，并在 Next.js 的 layout 层面为 studio 路由组单独定义一个无全局导航的布局。

### 2.3 与当前实现的差异

当前实现使用 `/studio` 路由，嵌套在 `(aistudio)` 布局组内。改造为全页模式需要：

- 将 studio 路由从 `(aistudio)` 组中独立出来，或在 studio 的 layout 中隐藏全局导航
- 画布容器从 `h-[calc(100vh-8rem)]` 改为 `h-screen w-screen`
- 增加明确的「返回」导航

## 3. 流程数据模型分析

### 3.1 当前数据模型的不足

当前画布数据以 `WorkflowSnapshot` 的 JSON 格式存储在 VFS 中：

```
VFS: /创作工坊/画布/{canvas_id}.json
内容: { version, canvasId, reactflow: { nodes, edges, viewport }, updatedAt }
```

这种纯文件存储方式存在以下局限：

- 无法按故事板、项目、用户维度查询画布
- 无法建立画布节点与后端实体（Task、Asset、Storyboard）的数据库级关联
- 无法支持后端调度（定时运行、批量运行）
- 无法记录执行历史和版本变更

### 3.2 建议的数据模型

创作工坊需要在数据库中建立自己的数据模型体系，核心包含三层：

**第一层：画布（Canvas）**

画布是创作工坊的顶层容器，代表一次完整的创作会话。

| 字段                | 类型                | 说明                                   |
| ------------------- | ------------------- | -------------------------------------- |
| id                  | UUID                | 主键                                   |
| name                | String              | 画布名称                               |
| project_id          | UUID (FK)           | 所属项目                               |
| episode_id          | UUID (FK, nullable) | 关联分集（可选）                       |
| user_id             | UUID (FK)           | 创建者                                 |
| status              | Enum                | draft / running / completed / archived |
| canvas_json_node_id | UUID (FK)           | VFS 中画布 JSON 文件的 FileNode ID     |
| node_count          | Integer             | 节点数量（冗余，便于列表展示）         |
| created_at          | DateTime            | 创建时间                               |
| updated_at          | DateTime            | 最后修改时间                           |

**第二层：工作流节点（CanvasNode）**

将画布中的关键节点（特别是 AI 生成节点）注册到数据库，建立与后端实体的关联。

| 字段                | 类型                | 说明                                         |
| ------------------- | ------------------- | -------------------------------------------- |
| id                  | UUID                | 主键                                         |
| canvas_id           | UUID (FK)           | 所属画布                                     |
| frontend_node_id    | String              | 前端 ReactFlow 节点 ID                       |
| node_type           | String              | 节点类型（generatorNode, slicerNode 等）     |
| storyboard_id       | UUID (FK, nullable) | 关联的故事板镜头                             |
| asset_id            | UUID (FK, nullable) | 关联的资产                                   |
| config_json         | JSONB               | 节点配置（提示词、模型、参数等）             |
| status              | Enum                | idle / queued / running / succeeded / failed |
| last_task_id        | UUID (FK, nullable) | 最近一次执行的任务 ID                        |
| output_file_node_id | UUID (FK, nullable) | 输出文件在 VFS 中的 FileNode ID              |
| created_at          | DateTime            | 创建时间                                     |

**第三层：执行记录（CanvasExecution）**

记录每次画布执行（单次运行或批量运行）的历史。

| 字段            | 类型                | 说明                                    |
| --------------- | ------------------- | --------------------------------------- |
| id              | UUID                | 主键                                    |
| canvas_id       | UUID (FK)           | 所属画布                                |
| trigger_type    | Enum                | manual / scheduled / batch              |
| status          | Enum                | running / succeeded / failed / canceled |
| total_nodes     | Integer             | 本次执行涉及的节点数                    |
| completed_nodes | Integer             | 已完成节点数                            |
| started_at      | DateTime            | 开始时间                                |
| finished_at     | DateTime (nullable) | 结束时间                                |
| result_summary  | JSONB               | 执行结果摘要                            |

### 3.3 数据流向

```
用户操作画布 → 前端 ReactFlow 状态
                    ↓ (debounce 保存)
              VFS JSON 文件（画布完整快照）
                    ↓ (关键节点同步)
              Canvas / CanvasNode 数据库记录
                    ↓ (执行时)
              Task 任务系统 → AI 网关 → 生成结果
                    ↓ (结果回写)
              VFS 文件（图片/视频/音频）
                    ↓ (关联)
              CanvasNode.output_file_node_id + Storyboard 关联
```

## 4. 任务系统对接与文件归属分析

### 4.1 当前任务系统架构

当前任务系统基于 Redis `brpop` 的自研队列，核心组件：

- **Task 模型：** 通用任务表，包含 type、status（queued/running/succeeded/failed/canceled）、progress（0-100）、input_json、result_json
- **Worker：** 单进程异步循环，从 Redis 队列取任务，按 type 分发到对应 Handler
- **Handler 注册表：** 19 种任务类型，涵盖分镜生成、资产提取、图片生成、视频生成等
- **实时推送：** WebSocket `/ws/tasks` 推送任务进度
- **前端队列：** BatchQueueManager 管理并发（默认 maxConcurrency=3），拓扑排序执行

### 4.2 创作工坊与任务系统的对接方案

创作工坊的 AI 生成节点需要通过任务系统执行，对接方式分为两层：

**前端层（已有，需增强）：**

- BatchQueueManager 负责前端并发控制和拓扑排序
- 每个 GeneratorNode 执行时调用 `/api/ai/image/generate` 或 `/api/ai/video/generate`
- 后端返回 task_id，前端通过 WebSocket 订阅进度

**后端层（需新增）：**

- 新增 `canvas_batch_execute` 任务类型，支持后端发起的批量执行
- 新增 `canvas_scheduled_execute` 任务类型，支持定时执行
- Canvas 和 CanvasNode 数据库记录与 Task 表通过 `entity_type='canvas_node'` + `entity_id` 关联

### 4.3 生成文件的归属与分类

创作工坊产出的媒体文件（图片、视频、音频）需要明确的归属关系和存储分类：

**VFS 目录结构建议：**

```
/项目根目录/
  └── 创作工坊/
        └── {canvas_name}/
              ├── 图片/
              │     ├── SH01_角色设计_v1.png
              │     └── SH02_场景_v1.png
              ├── 视频/
              │     ├── SH01_动画_v1.mp4
              │     └── SH02_动画_v1.mp4
              └── 音频/
                    └── SH01_配音_v1.wav
```

**文件命名规则：** `{镜头编号}_{用途描述}_v{版本号}.{扩展名}`

**归属关系链：**

- 文件 → FileNode（VFS 记录）
- FileNode → CanvasNode.output_file_node_id（画布节点关联）
- CanvasNode → Storyboard（故事板镜头关联）
- 文件同时可通过 AssetResource 关联到资产变体（用户主动「保存为资产」时）

**分类规则：**

| 生成类型       | VFS 子目录 | MinIO Bucket     | 关联实体                        |
| -------------- | ---------- | ---------------- | ------------------------------- |
| AI 生成图片    | 图片/      | anyreason-assets | CanvasNode + 可选 AssetResource |
| AI 生成视频    | 视频/      | anyreason-assets | CanvasNode + 可选 VideoPrompt   |
| AI 生成音频    | 音频/      | anyreason-assets | CanvasNode                      |
| 用户上传参考图 | 参考/      | anyreason-assets | CanvasNode                      |

### 4.4 与故事板的挂载机制

创作工坊产出的媒体文件需要能够「挂载」到故事板镜头上，实现方式：

1. **自动挂载：** 当 GeneratorNode 关联了某个 Storyboard 记录（通过上游 StoryboardNode 的 sourceStoryboardId），生成成功后自动将 output_file_node_id 写入该 Storyboard 的 active_assets JSONB 字段
2. **手动挂载：** 用户在画布中选择生成结果，右键「关联到镜头」，选择目标镜头后建立关联
3. **批量挂载：** 批量执行完成后，按照 StoryboardNode → GeneratorNode 的连线关系，自动将所有生成结果挂载到对应镜头

## 5. 节点体系语义审查

### 5.1 当前 10 种节点的逐一分析

#### 创作组（Creation）

| 节点                                 | 设计意图                                       | 问题                                                                           | 建议                                                                       |
| ------------------------------------ | ---------------------------------------------- | ------------------------------------------------------------------------------ | -------------------------------------------------------------------------- |
| **TextNoteNode（文本笔记）**   | 用户在画布上记录自由文本笔记                   | 定位清晰，无问题                                                               | 保留，作为辅助工具                                                         |
| **ScriptNode（剧本节点）**     | 承载剧本文本，作为拆分节点的输入源             | 与右侧「故事板」Tab 的剧本选择器功能重叠；用户需要手动复制粘贴剧本文本到节点中 | 建议改造为「剧本引用节点」，自动从后端拉取选定分集的剧本文本，而非手动输入 |
| **StoryboardNode（分镜节点）** | 代表一个具体的镜头，包含场景描述、对白、参考图 | 这是创作工坊最核心的节点，定位正确                                             | 保留并强化，增加与后端 Storyboard 记录的双向同步                           |

#### AI 生成组（AI Generation）

| 节点                                | 设计意图                     | 问题                                                                                                               | 建议                                               |
| ----------------------------------- | ---------------------------- | ------------------------------------------------------------------------------------------------------------------ | -------------------------------------------------- |
| **GeneratorNode（生成节点）** | 调用 AI 生成图片或视频       | 这是创作工坊的核心生产力节点，定位正确。但当前缺少参考图输入的可视化和提示词编辑的完整 UI                          | 保留并强化，增加参考图预览、提示词模板、模型选择器 |
| **SlicerNode（拆分节点）**    | 将剧本文本 AI 拆分为分镜列表 | 功能有价值，但使用场景有限——大多数情况下用户已经在后端完成了分镜拆分（通过 episode_storyboard_agent_apply 任务） | 保留但降低优先级，作为画布内的辅助工具             |
| **CandidateNode（提取节点）** | 从文本中 AI 提取资产候选     | 同上，后端已有 4 种资产提取任务类型                                                                                | 保留但降低优先级                                   |

#### 展示组（Display）

| 节点                              | 设计意图                      | 问题                                                                                                                                                                                                                            | 建议                                                                                                                                                 |
| --------------------------------- | ----------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------- |
| **PreviewNode（预览节点）** | 展示图片或视频的预览          | **语义模糊。** GeneratorNode 自身已有结果预览能力（显示 lastImage），PreviewNode 的独立存在价值不明确。在自动生成的工作流中，每个 Generator 下方都会创建一个 Preview，但 Preview 只是被动接收上游图片，没有额外功能       | **建议合并到 GeneratorNode。** Generator 生成结果后直接在自身展示预览，无需额外的 Preview 节点。如果需要独立的「画廊」功能，可以用右侧面板实现 |
| **MediaNode（媒体节点）**   | 展示静态媒体文件（图片/视频） | **与 GeneratorNode 和 PreviewNode 职责重叠。** 在早期设计中 MediaNode 是 LLM 节点（有提示词面板），但在统一节点体系后被归入展示组，变成了一个没有 AI 能力的静态展示节点。与 PreviewNode 的区别仅在于 MediaNode 有输出端口 | **建议合并。** 如果需要「引入外部媒体文件」的能力，可以作为 GeneratorNode 的一种模式（手动上传模式 vs AI 生成模式）                            |

#### 引用组（Reference）

| 节点                                | 设计意图                                                          | 问题                                                                                                                                                                                                                                      | 建议                                                                                                        |
| ----------------------------------- | ----------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------- |
| **AssetNode（资产节点）**     | 引用项目中的资产（角色/场景/道具/特效），提供资产缩略图和变体选择 | 定位清晰——将资产库中的资产拖入画布，作为 GeneratorNode 的参考输入                                                                                                                                                                       | 保留                                                                                                        |
| **ReferenceNode（引用节点）** | 引用故事板中的镜头条目（shot_code、描述、对白）                   | **与 StoryboardNode 语义高度重叠。** ReferenceNode 从右侧故事板 Tab 拖入，包含 shot_code 和描述；StoryboardNode 也从右侧故事板 Tab 拖入，同样包含 shot_code 和描述。两者的区别仅在于 StoryboardNode 有端口定义而 ReferenceNode 没有 | **建议合并到 StoryboardNode。** 所有从故事板拖入的条目统一创建 StoryboardNode，ReferenceNode 可以废弃 |

### 5.2 关于「引用组」与右侧资产库的关系

用户提出的疑问完全合理：右侧有可拖拽的资产库，画布上又有「引用组」下的 AssetNode 和 ReferenceNode，这三者的关系确实令人困惑。

**澄清：**

- 右侧资产库 Tab 是资产的「来源面板」，用户从这里拖拽资产到画布上
- 拖入画布后创建的就是 AssetNode（资产节点）
- AssetNode 的作用是在画布上「代表」一个资产，可以通过连线将其作为 GeneratorNode 的参考输入
- ReferenceNode 的原始设计意图是「引用故事板镜头条目」，但这个功能已经被 StoryboardNode 覆盖

**建议的简化方案：**

- 引用组只保留 AssetNode（资产引用）
- 故事板条目统一使用 StoryboardNode（创作组）
- ReferenceNode 废弃

### 5.3 建议的精简节点体系

经过审查，建议将 10 种节点精简为 7 种：

| 组别    | 节点           | 用途                                              |
| ------- | -------------- | ------------------------------------------------- |
| 创作    | TextNoteNode   | 自由文本笔记                                      |
| 创作    | ScriptNode     | 剧本文本引用（改为自动拉取）                      |
| 创作    | StoryboardNode | 故事板镜头（合并原 ReferenceNode）                |
| AI 生成 | GeneratorNode  | AI 图片/视频生成（合并原 PreviewNode 的预览能力） |
| AI 生成 | SlicerNode     | AI 剧本拆分                                       |
| AI 生成 | CandidateNode  | AI 资产提取                                       |
| 引用    | AssetNode      | 资产引用                                          |

被合并/废弃的节点：

- ~~PreviewNode~~ → 合并到 GeneratorNode
- ~~MediaNode~~ → 合并到 GeneratorNode（手动上传模式）
- ~~ReferenceNode~~ → 合并到 StoryboardNode

## 6. 生成节点的参考与提示词机制

### 6.1 当前参考图机制

当前 GeneratorNode 通过端口连线接收参考输入：

- `in-script`（text 类型）：接收上游 StoryboardNode 的场景描述文本
- `in-ref`（asset-ref 类型）：接收上游 AssetNode 的资产引用

但实际的参考图传递存在断层：

- AssetNode 只存储 assetId 和 name，没有实际的图片 URL
- 连线建立的是逻辑关系，但生成请求中的 `images[]` 参数需要实际的图片数据
- 当前 executeTask 回调只传递了 prompt、negPrompt、resolution，没有传递参考图

### 6.2 建议的参考图机制

**参考图来源优先级：**

1. **AssetNode 连线：** 通过 AssetNode 的 assetId 查询后端获取资产的代表图（AssetResource 中 is_default 的图片），转为 data URL 后加入 `images[]`
2. **其他 GeneratorNode 连线：** 如果上游 GeneratorNode 已有生成结果（lastImage），将其作为参考图
3. **用户手动上传：** 在 GeneratorNode 的提示词面板中增加「上传参考图」按钮，用户可以直接上传本地图片
4. **StoryboardNode 的参考图：** 如果 StoryboardNode 有 referenceImageUrl，也可以作为参考

**参考图收集流程：**

- 当用户点击「生成」时，前端遍历所有指向该 GeneratorNode 的 incoming edges
- 按来源类型收集可用的图片 URL
- 将图片下载并转为 data URL（避免跨域问题）
- 打包到生成请求的 `images[]` 参数中
- 限制：最多 4 张参考图，单张不超过 4MB

### 6.3 提示词编辑机制

当前 GeneratorNode 的提示词编辑能力非常有限——只在节点内显示一行 prompt 文本，没有编辑入口。

**建议的提示词编辑方案：**

**方案：节点 Focus 态展开提示词面板**

当用户选中 GeneratorNode 且 zoom ≥ 0.6 时，在节点下方或右侧展开一个提示词编辑面板（屏幕空间浮层，不随缩放变化）：

面板内容：

- 参考来源预览区（横向滚动的缩略图列表，来自连线收集）
- 正向提示词输入框（多行文本，支持提示词模板）
- 负面提示词输入框
- 模型选择下拉框（从 AIModelConfig 拉取可用模型列表）
- 画幅/比例选择（1:1, 16:9, 9:16, 4:3 等）
- 生成模式切换（图片/视频）
- 视频时长选择（仅视频模式）
- 「生成」按钮

这个方案实际上在早期设计文档（方案A）中已经有详细描述（LlmPromptPanel），当前代码中也有 `LlmPromptPanel` 组件的骨架实现，但它只服务于 MediaNode（kind='media'）。需要将其迁移到 GeneratorNode 上。

### 6.4 提示词模板

为了提高效率，建议支持提示词模板功能：

- 系统预设模板（如「角色立绘」「场景全景」「动作特写」等）
- 用户自定义模板（复用现有的 AIPromptPreset 表）
- 模板变量替换（如 `{角色名}` `{场景描述}` 自动从连线的 AssetNode/StoryboardNode 中提取）

## 7. 任务系统并发与多人协作能力评估

### 7.1 当前并发能力

**后端 Worker：**

- 单进程异步循环，通过 `brpop` 从 Redis 队列逐个取任务
- 每个任务在 asyncio 事件循环中执行，理论上支持 I/O 并发（等待 AI API 响应时不阻塞其他任务）
- 但实际上 `_worker_loop` 是串行取任务的：取一个 → 处理完 → 取下一个
- 任务超时控制：图片生成 5 分钟，视频生成 30 分钟，其他 10 分钟

**前端 BatchQueueManager：**

- maxConcurrency 默认 3，即同时最多 3 个节点在执行
- 拓扑排序保证依赖关系
- 超时检测（默认 300 秒）

**实际并发瓶颈：**

| 层级         | 并发能力                          | 瓶颈               |
| ------------ | --------------------------------- | ------------------ |
| 前端队列     | 3 个并发请求                      | 可配置，不是瓶颈   |
| Next.js BFF  | 无限制（受 Node.js 事件循环限制） | 不是瓶颈           |
| FastAPI 后端 | 异步框架，理论上高并发            | 不是瓶颈           |
| Redis 队列   | 单队列，brpop 串行消费            | **主要瓶颈** |
| Worker 进程  | 单进程，串行取任务                | **主要瓶颈** |
| AI API 调用  | 受厂商 QPS 限制                   | 外部瓶颈           |

**结论：** 当前架构下，后端 Worker 是串行处理任务的。虽然单个任务内部是异步的（等待 AI API 时不阻塞 CPU），但 Worker 不会同时处理多个任务。这意味着：

- 如果有 10 个图片生成任务排队，它们会一个接一个执行
- 前端的 maxConcurrency=3 只是控制「同时发起请求」的数量，但后端仍然串行处理
- 实际吞吐量取决于单个 AI API 调用的耗时

### 7.2 多人同时工作的能力

**当前不支持多人协作。** 具体表现：

- 画布数据存储在 VFS JSON 文件中，没有并发控制
- 如果两个用户同时编辑同一个画布，后保存的会覆盖先保存的（Last Write Wins）
- 没有实时同步机制（如 CRDT 或 OT）
- 没有画布级别的锁定机制

**多人场景下的风险：**

- 数据丢失：两人同时编辑，一人的修改被覆盖
- 任务冲突：两人同时对同一个节点发起生成，产生重复任务
- 状态不一致：一人看到的节点状态与另一人不同

### 7.3 提升并发能力的建议

**短期（Worker 多实例）：**

- 启动多个 Worker 进程，每个进程独立从 Redis 队列取任务
- Redis `brpop` 天然支持多消费者竞争消费，无需修改队列逻辑
- 建议启动 3-5 个 Worker 进程，与前端 maxConcurrency 匹配
- 实现方式：在 docker-compose 中将 worker 服务的 `replicas` 设为 3-5

**中期（任务优先级队列）：**

- 引入多个 Redis 队列，按优先级区分（如 high/normal/low）
- 图片生成走 high 队列（用户等待中），批量任务走 normal 队列
- Worker 优先消费 high 队列

**长期（多人协作）：**

- 画布数据从 VFS JSON 迁移到数据库（Canvas + CanvasNode 模型）
- 引入 WebSocket 实时同步（基于操作日志的 OT 或 CRDT）
- 画布级别的乐观锁或悲观锁
- 这是一个重大架构变更，建议作为独立里程碑规划

## 8. 导出与剪辑工作流设计

### 8.1 导出需求分析

创作工坊的最终产出是一系列按镜头组织的媒体文件（图片、视频、音频），用户需要将这些产出导出到专业剪辑软件（如 Premiere Pro、DaVinci Resolve、Final Cut Pro）中进行后期制作。

导出的核心挑战：

- 如何保持镜头的顺序和结构信息
- 如何让剪辑软件能够直接识别和使用这些素材
- 如何减少用户在剪辑软件中的重复性操作

### 8.2 建议的导出方案

**层级一：素材包导出（基础）**

将画布中所有生成结果按镜头顺序打包为 ZIP 文件：

```
导出包/
  ├── manifest.json          # 元数据（镜头列表、时长、资产关联）
  ├── SC01/
  │     ├── SH01_image.png   # 镜头 1 的图片
  │     ├── SH01_video.mp4   # 镜头 1 的视频
  │     └── SH01_audio.wav   # 镜头 1 的音频（如有）
  ├── SC02/
  │     ├── SH02_image.png
  │     └── SH02_video.mp4
  └── assets/
        ├── 张三_角色设计.png  # 使用到的资产
        └── 李四_角色设计.png
```

**层级二：时间线 XML 导出（进阶）**

生成 FCP XML（Final Cut Pro XML Interchange Format）或 AAF 格式的时间线文件，剪辑软件可以直接导入：

- 按镜头顺序排列视频片段
- 自动设置每个片段的入点和出点
- 保留镜头编号和场景描述作为标记（Marker）
- 音频轨道独立（如有配音）

FCP XML 是目前兼容性最好的跨软件时间线格式，Premiere Pro、DaVinci Resolve、Final Cut Pro 都支持导入。

**层级三：在线预览与标注（远期）**

在平台内提供简易的时间线预览功能：

- 按镜头顺序播放所有视频片段
- 支持添加文字标注和修改意见
- 支持导出带标注的审片报告
- 这个功能可以减少「导出 → 剪辑软件 → 审片 → 反馈」的循环次数

### 8.3 导出流程设计

```
用户点击「导出」
    ↓
选择导出格式（素材包 / FCP XML / 在线预览）
    ↓
选择导出范围（全部镜头 / 选中镜头 / 指定场次）
    ↓
后端创建导出任务（新增 canvas_export 任务类型）
    ↓
Worker 执行：
  1. 从 VFS/MinIO 收集所有相关媒体文件
  2. 按格式要求组织文件结构
  3. 生成 manifest.json 或 FCP XML
  4. 打包为 ZIP 并上传到 MinIO exports bucket
    ↓
返回下载链接，用户下载
```

### 8.4 与现有系统的集成

- 导出任务复用现有 Task 系统，新增 `canvas_export` 任务类型和对应 Handler
- 导出文件存储在 MinIO 的 `anyreason-exports` bucket（已存在）
- 导出记录可以关联到 Canvas 和 CanvasExecution

## 总结

创作工坊当前面临的核心问题不是技术实现的缺陷，而是产品定位和架构设计的模糊。通过明确「创作工坊的工作对象是故事板」这一核心定位，可以推导出一系列清晰的设计决策：

- 全页沉浸式界面，因为故事板创作需要最大化的画布空间
- 精简节点体系（10 → 7），因为故事板创作不需要那么多语义重叠的节点
- 数据库级的数据模型，因为故事板创作的产出需要与项目、分集、镜头建立结构化关联
- 后端化的任务调度，因为批量生成和定时执行是故事板生产流程的刚需
- 标准化的导出格式，因为故事板的最终归宿是专业剪辑软件

建议按 M1 → M2 → M3 → M4 → M5 的顺序推进，每个里程碑都是可独立交付的增量。M1 是最紧迫的，解决了用户当前最直接的困惑（界面受限、节点混乱、提示词无法编辑）。M2 是架构基础，为后续所有功能提供数据支撑。M3-M5 可以根据业务优先级灵活调整顺序。
