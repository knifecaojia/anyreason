## 1. 现状回顾与问题诊断

### 1.1 当前状态

创作工坊目前已实现的能力：

- 基于 ReactFlow 的无限画布，点状网格背景
- 10 种节点类型，分为 4 组（创作、AI 生成、展示、引用）
- 左侧悬浮菜单 + 右侧可折叠 Tab Pages（故事板/资产库/详情）
- VFS 持久化（画布 JSON 存储在 `/创作工坊/画布/` 目录下）
- 批量执行队列（BatchQueueManager，前端拓扑排序 + 并发控制）
- 一键生成工作流（从分集数据自动创建 Script → Slicer → Storyboard → Generator → Preview 链路；重构后目标链路为 Script → Slicer → Storyboard → **TextGenNode** → Generator，详见 5.3/5.4 节）
- 节点间数据流传播（Kahn 拓扑排序 + 端口类型匹配）
- 撤销/重做、复制/粘贴、对齐辅助线
- 导入/导出画布 JSON

### 1.2 核心问题

经过对代码和设计文档的全面审查，当前创作工坊存在以下结构性问题：

**问题一：设计哲学缺失，借鉴过度。** 当前 UI/UE 大量借鉴 Tapnow 和开源项目的布局模式，但 Tapnow 是通用图片/视频生成工具，而本项目的画布需要服务于动画生产流程中的多种创作场景——从故事板可视化、到角色设计迭代、到场景氛围探索。画布作为独立创作单元，需要在通用画布能力的基础上，提供对剧本/故事板/资产等领域数据的**按权限引用**能力，而非简单复制通用工具的交互范式。

**问题二：节点语义混乱。** 10 种节点类型中存在职责重叠和语义模糊：

- AssetNode（资产节点）和 ReferenceNode（引用节点）都在「引用组」，但用途不同却缺乏清晰区分
- MediaNode（展示组）和 GeneratorNode（AI 生成组）都涉及图片/视频，但一个是静态展示，一个是 AI 生成，用户难以理解何时用哪个
- PreviewNode 的存在意义不明确——GeneratorNode 自身已有结果预览能力

**问题三：数据模型缺乏后端支撑。** 画布数据完全以 JSON 文件形式存储在 VFS 中，没有独立的数据库模型。这意味着：

- 无法对画布/流程进行结构化查询
- 无法建立画布与故事板、资产、任务之间的数据库级关联
- 无法支持定时运行、批量运行等需要后端调度的场景

**问题四：画布定位需要明确为独立概念。** 设计文档早期声称「画布不与剧本/剧集强绑定」，这一方向是正确的——**画布应当作为系统中一个独立的、自治的创作单元**，不从属于任何特定剧本或剧集。创作人员在创意工坊的全屏界面中对画布进行增删改查，画布创作过程中根据用户权限来获取剧本、剧集、故事板和资产作为参考素材。这意味着：

- 画布不再绑定 episode_id，一个画布可以引用多个剧集的故事板
- 剧本、故事板、资产是画布创作的「输入参考」，而非画布的「所属实体」
- 画布的产出（图片、视频等）独立存储在画布自身的 VFS 目录下，仅在用户主动「导出到故事板」时才建立关联
- 权限模型决定了创作者能在画布中引用哪些项目的剧本和资产

## 2. 全页界面设计分析

### 2.1 当前布局的问题

当前创作工坊嵌套在 `(aistudio)` 路由组的通用布局中，画布区域被限制在 `h-[calc(100vh-8rem)]` 的容器内，顶部有全局导航栏占据空间。对于一个以画布为核心的创作工具，这种布局浪费了宝贵的屏幕空间。

### 2.2 全页界面方案

创作工坊应该采用全屏沉浸式布局，包含两个层级：**画布列表页**和**画布编辑页**。

#### 2.2.1 画布列表页（创意工坊入口）

创作人员进入创意工坊后，首先看到的是**画布列表页**（`/studio`），支持对画布进行增删改查：

**列表展示：**

- 卡片式布局，每张卡片显示：画布名称、缩略图（最近一次快照）、创建时间、更新时间、节点数量、状态标签（draft/active/archived）
- 支持按创建时间、更新时间、名称排序
- 支持按状态筛选（全部 / 活跃 / 草稿 / 归档）
- 搜索框：按画布名称模糊搜索

**CRUD 操作：**

| 操作 | 触发方式                 | 说明                                                         |
| ---- | ------------------------ | ------------------------------------------------------------ |
| 创建 | 顶部「新建画布」按钮     | 弹出对话框输入画布名称，创建后直接跳转到编辑页               |
| 查看 | 点击卡片                 | 跳转到 `/studio/[canvasId]` 编辑页                           |
| 编辑 | 卡片右上角菜单「重命名」 | 行内编辑画布名称                                             |
| 删除 | 卡片右上角菜单「删除」   | 二次确认后软删除（status → archived），可在归档列表中恢复    |
| 复制 | 卡片右上角菜单「复制」   | 深拷贝画布 JSON 和数据库记录，生成新画布                     |

**路由设计：** `/studio` 为列表页，`/studio/[canvasId]` 为编辑页，均在独立的 studio 路由组中，无全局导航栏。

#### 2.2.2 画布编辑页（全屏沉浸模式）

**入口方式：** 从画布列表页点击卡片进入，或从其他页面（如故事板列表）通过「在创意工坊中打开」快捷入口进入。以全屏沉浸模式呈现，类似 Figma 编辑器。

**布局结构：**

- 画布占据整个浏览器窗口（100vw × 100vh）
- 顶部极简工具栏（高度 40px 左右），包含：返回按钮（回到列表页）、画布名称（可点击编辑）、保存状态、用户头像
- 左侧工具面板（可折叠）：节点面板、模板面板
- 右侧数据面板（可折叠）：**参考数据浏览器**（见 2.2.3）
- 没有全局导航栏、没有侧边菜单栏

**退出方式：** 顶部左上角的返回按钮，或 ESC 键，返回到画布列表页。

#### 2.2.3 参考数据浏览器（右侧面板）

画布作为独立概念，不绑定特定剧本或剧集。创作者在画布中需要引用的剧本、故事板、资产等数据，通过右侧面板的**参考数据浏览器**按权限获取：

**Tab 页结构：**

| Tab        | 内容                                                                                       | 数据来源                         |
| ---------- | ------------------------------------------------------------------------------------------ | -------------------------------- |
| 剧本       | 用户有权限访问的项目下的剧本列表，点击可预览全文，拖拽可创建 ScriptNode                    | `GET /api/scripts?project_id=`   |
| 故事板     | 用户有权限访问的分集下的故事板列表，支持按分集/场次筛选，拖拽可创建 StoryboardNode         | `GET /api/storyboards?episode_id=` |
| 资产库     | 用户有权限访问的项目资产，支持按类型/标签筛选，拖拽可创建 AssetNode                        | `GET /api/assets?project_id=`    |
| 画布详情   | 当前画布的元信息、节点统计、执行历史                                                       | 本地状态 + Canvas API            |

**权限模型：**

- 画布本身属于创建者（`user_id`），但**不绑定特定项目**
- 创作者能浏览和引用的剧本/故事板/资产范围，由其在各项目中的角色权限决定（复用现有的 `ProjectMember` 权限体系）
- 如果创作者有多个项目的权限，参考数据浏览器中可切换项目上下文
- 画布本身的访问控制：当前阶段仅创建者可见，后续 M4+ 可扩展为团队共享

**路由设计：** 使用 `/studio/[canvasId]` 作为路由，并在 Next.js 的 layout 层面为 studio 路由组单独定义一个无全局导航的布局。

### 2.3 与当前实现的差异

当前实现使用 `/studio` 路由，嵌套在 `(aistudio)` 布局组内，且没有画布列表页。改造为两级全页模式需要：

- 将 studio 路由从 `(aistudio)` 组中独立出来，建立独立的 layout（无全局导航）
- 新增 `/studio` 画布列表页（卡片式 CRUD 界面）
- `/studio/[canvasId]` 画布编辑页，画布容器从 `h-[calc(100vh-8rem)]` 改为 `h-screen w-screen`
- 右侧面板从「固定绑定的故事板/资产」改为「基于权限的参考数据浏览器」
- 去除画布与 episode_id 的绑定逻辑，改为用户自由选择引用源

## 3. 流程数据模型分析

### 3.1 当前数据模型的不足

当前画布数据以 `WorkflowSnapshot` 的 JSON 格式存储在 VFS 中：

```
VFS: /创作工坊/画布/{canvas_id}.json
内容: { version, canvasId, reactflow: { nodes, edges, viewport }, updatedAt }
```

这种纯文件存储方式存在以下局限：

- 无法按用户、状态等维度查询和管理画布（增删改查）
- 无法建立画布节点与后端实体（Task、Asset、Storyboard）的数据库级关联
- 无法支持后端调度（定时运行、批量运行）
- 无法记录执行历史和版本变更

### 3.2 建议的数据模型

创作工坊需要在数据库中建立自己的数据模型体系，核心包含三层：

**第一层：画布（Canvas）**

画布是创作工坊的顶层容器，代表一次独立的创作会话。**画布不绑定特定项目或分集**，是用户级别的独立资源。

| 字段                | 类型                | 说明                                   |
| ------------------- | ------------------- | -------------------------------------- |
| id                  | UUID                | 主键                                   |
| name                | String              | 画布名称                               |
| user_id             | UUID (FK)           | 创建者（画布的所有者）                 |
| description         | String (nullable)   | 画布描述（可选，便于列表展示和搜索）   |
| status              | Enum                | draft / active / archived（注：无 running/completed，画布的「执行中」状态由 CanvasExecution 跟踪） |
| canvas_json_node_id | UUID (FK)           | VFS 中画布 JSON 文件的 FileNode ID     |
| thumbnail_node_id   | UUID (FK, nullable) | 画布缩略图在 VFS 中的 FileNode ID（用于列表页卡片展示） |
| node_count          | Integer             | 节点数量（冗余，便于列表展示）         |
| created_at          | DateTime            | 创建时间                               |
| updated_at          | DateTime            | 最后修改时间                           |

> **⚠️ 画布独立化变更：** 原设计中 Canvas 表包含 `project_id`（FK，必填）和 `episode_id`（FK，可选）字段，将画布与项目/分集绑定。现改为**去除这两个外键**，画布不再从属于任何项目或分集。画布中引用的剧本、故事板、资产通过节点内的 `config_json` 记录来源 ID（如 `sourceScriptId`、`sourceStoryboardId`、`sourceAssetId`），这些是**弱引用**（不设 FK 约束），即使源数据被删除也不影响画布本身。
>
> **Alembic 迁移影响：** 需要新增迁移脚本，从 `canvases` 表中 DROP `project_id` 和 `episode_id` 列及其外键约束和索引。

**第二层：工作流节点（CanvasNode）**

将画布中的**所有 8 种节点类型**注册到数据库（CHECK 约束涵盖全部类型，见 5.4.7 节）。非 AI 节点（TextNoteNode、ScriptNode 等）的 status 保持 `pending` 不变，仅 AI 生成节点会经历完整的状态流转。

| 字段                | 类型                | 说明                                         |
| ------------------- | ------------------- | -------------------------------------------- |
| id                  | UUID                | 主键                                         |
| canvas_id           | UUID (FK)           | 所属画布                                     |
| frontend_node_id    | String(64)          | 前端 ReactFlow 节点 ID（UNIQUE with canvas_id） |
| node_type           | String(32)          | 节点类型（CHECK 约束，见 5.4.7 节）          |
| source_storyboard_id | UUID (nullable)    | 引用的故事板镜头 ID（**弱引用，无 FK 约束**，源数据删除不影响节点） |
| source_asset_id     | UUID (nullable)     | 引用的资产 ID（**弱引用，无 FK 约束**）      |
| config_json         | JSONB               | 节点**静态**配置（提示词模板、模型、参数等，不含 lastOutput 等运行时数据，见 3.4 节） |
| status              | Enum                | pending / running / completed / failed |
| last_task_id        | UUID (FK, nullable) | 最近一次执行的任务 ID（TextGenNode 不使用，因其不经过任务系统） |
| output_file_node_id | UUID (FK, nullable) | 输出文件在 VFS 中的 FileNode ID（TextGenNode 不使用，因其不产出文件） |
| created_at          | DateTime            | 创建时间                                     |
| updated_at          | DateTime            | 最后修改时间                                 |

> **弱引用说明：** `source_storyboard_id` 和 `source_asset_id` 不设外键约束（无 `FOREIGN KEY`），仅作为记录节点数据来源的标识。这是因为画布是独立概念，引用的故事板或资产可能属于不同项目，且源数据被删除后画布节点应继续正常工作（节点内已保存了快照数据）。查询时通过应用层 JOIN 而非数据库外键实现关联。

**第三层：执行记录（CanvasExecution）**

记录每次画布执行（单次运行或批量运行）的历史。

| 字段            | 类型                | 说明                                    |
| --------------- | ------------------- | --------------------------------------- |
| id              | UUID                | 主键                                    |
| canvas_id       | UUID (FK)           | 所属画布                                |
| trigger_type    | Enum                | manual / batch / auto                   |
| status          | Enum                | pending / running / completed / partial / failed / cancelled |
| total_nodes     | Integer             | 本次执行涉及的节点数                    |
| completed_nodes | Integer             | 已完成节点数                            |
| started_at      | DateTime            | 开始时间                                |
| finished_at     | DateTime (nullable) | 结束时间                                |
| result_summary  | JSONB               | 执行结果摘要                            |

### 3.3 数据流向

```
用户操作画布 → 前端 ReactFlow 状态
                    ↓ (debounce 保存)
              VFS JSON 文件（画布完整快照）——Source of Truth
                    ↓ (关键节点同步)
              Canvas / CanvasNode 数据库记录——查询索引
                    ↓ (执行时)
              Task 任务系统 → AI 网关 → 生成结果
                    ↓ (结果回写)
              VFS 文件（图片/视频/音频）→ CanvasNode.output_file_node_id
                    ↓ (用户主动导出时，可选)
              导出到故事板 / 导出为资产（见 4.4 节）
```

### 3.4 VFS 与数据库的双写一致性策略

画布数据同时存在于 VFS JSON 文件和数据库两处，必须明确一致性规则，否则长期运行后二者会产生漂移。

**核心原则：VFS JSON 是 Source of Truth（权威数据源），数据库是查询索引（投影）。**

| 维度         | VFS JSON 文件                              | 数据库（Canvas / CanvasNode）                |
| ------------ | ------------------------------------------ | -------------------------------------------- |
| 存储内容     | 画布完整快照（所有节点、连线、视口、数据） | 画布元信息 + 关键节点的结构化字段            |
| 更新频率     | 高（用户每次操作后 debounce 1-3 秒写入）   | 低（关键事件触发：创建/删除节点、执行完成）  |
| 打开画布时   | 从 VFS JSON 加载全量数据                   | 仅用于画布列表页展示，不参与画布编辑         |
| 读写角色     | 前端读写，后端只读                         | 前端写、后端读写                             |

**一致性保障机制：**

1. **单向同步（VFS → DB）：** 前端在以下时机将 VFS JSON 中的关键信息同步到数据库：
   - 新增/删除 AI 生成节点时 → 创建/删除 CanvasNode 记录
   - 节点执行完成时 → 更新 CanvasNode.status、last_task_id、output_file_node_id
   - 画布保存时 → 更新 Canvas.node_count、updated_at
2. **幂等写入：** CanvasNode 通过 `UNIQUE(canvas_id, frontend_node_id)` 约束实现 upsert，重复同步不会产生冗余记录
3. **数据库不反写 VFS：** 避免双向同步导致的冲突。后端只通过数据库记录来索引和查询画布，不修改 VFS JSON
4. **启动校验（M3.4）：** 用户打开画布时，前端对比 VFS JSON 中的节点列表与数据库 CanvasNode 记录，如有偏差则以 VFS JSON 为准进行修复同步
5. **运行时数据分离：** `lastOutput`（TextGenNode 的 LLM 输出）等频繁变化的运行时数据只存储在 VFS JSON 的 `node.data` 中，**不写入** CanvasNode.config_json。config_json 仅存储静态配置（systemPrompt、bindingKey、presetId 等）

**不处理的场景（明确排除）：**

- 多用户同时编辑同一画布的冲突（见 7.2 节，当前不支持多人协作）
- VFS 文件损坏的恢复（依赖 MinIO 的对象存储可靠性）

## 4. 任务系统对接与文件归属分析

### 4.1 当前任务系统架构

当前任务系统基于 Redis `brpop` 的自研队列，核心组件：

- **Task 模型：** 通用任务表，包含 type、status（queued/running/succeeded/failed/canceled）、progress（0-100）、input_json、result_json
- **Worker：** 单进程异步循环，从 Redis 队列取任务，按 type 分发到对应 Handler
- **Handler 注册表：** 19 种任务类型，涵盖分镜生成、资产提取、图片生成、视频生成等
- **实时推送：** WebSocket `/ws/tasks` 推送任务进度
- **前端队列：** BatchQueueManager 管理并发（默认 maxConcurrency=3），拓扑排序执行

### 4.2 创作工坊与任务系统的对接方案

创作工坊的 AI 生成节点需要通过任务系统执行，对接方式分为两层：

**前端层（已有，需增强）：**

- BatchQueueManager 负责前端并发控制和拓扑排序
- 每个 GeneratorNode 执行时调用 `/api/ai/image/generate` 或 `/api/ai/video/generate`
- 后端返回 task_id，前端通过 WebSocket 订阅进度

**后端层（需新增）：**

- 新增 `canvas_batch_execute` 任务类型，支持后端发起的批量执行
- 新增 `canvas_scheduled_execute` 任务类型，支持定时执行
- Canvas 和 CanvasNode 数据库记录与 Task 表通过 `entity_type='canvas_node'` + `entity_id` 关联

### 4.3 生成文件的归属与分类

创作工坊产出的媒体文件（图片、视频、音频）**独立归属于画布**，不自动关联到任何项目或故事板。

**VFS 目录结构建议：**

```
/用户根目录/
  └── 创作工坊/
        └── {canvas_id}/
              ├── 画布.json              ← 画布快照（Source of Truth）
              ├── 图片/
              │     ├── node_{frontend_id}_v1.png
              │     └── node_{frontend_id}_v2.png
              ├── 视频/
              │     └── node_{frontend_id}_v1.mp4
              └── 音频/
                    └── node_{frontend_id}_v1.wav
```

> **目录变更说明：** 原设计使用 `/项目根目录/创作工坊/{canvas_name}/` 路径，隐含了画布从属于项目的关系。现改为 `/用户根目录/创作工坊/{canvas_id}/`，画布文件直接在用户的 VFS 空间下组织，不依赖项目目录结构。文件命名改为基于 `frontend_node_id`（节点级唯一）而非镜头编号，因为画布可能不包含任何故事板引用。

**文件命名规则：** `node_{frontend_node_id}_v{版本号}.{扩展名}`

**归属关系链：**

- 文件 → FileNode（VFS 记录）
- FileNode → CanvasNode.output_file_node_id（画布节点关联）
- （仅在用户主动导出时）→ 故事板关联 或 AssetResource（见 4.4 节）

**分类规则：**

| 生成类型       | VFS 子目录 | MinIO Bucket     | 关联实体                        |
| -------------- | ---------- | ---------------- | ------------------------------- |
| AI 生成图片    | 图片/      | anyreason-assets | CanvasNode（画布内部归属）      |
| AI 生成视频    | 视频/      | anyreason-assets | CanvasNode（画布内部归属）      |
| AI 生成音频    | 音频/      | anyreason-assets | CanvasNode（画布内部归属）      |
| 用户上传参考图 | 参考/      | anyreason-assets | CanvasNode（画布内部归属）      |

### 4.4 导出到故事板 / 资产（用户主动操作）

画布作为独立创作空间，其产出**不自动关联**到任何故事板或资产。用户在画布中完成创作后，可以**主动选择**将成果导出到目标故事板或保存为资产：

**导出方式：**

1. **单节点导出：** 用户右键点击 GeneratorNode，选择「导出到故事板」→ 弹出故事板选择器（按权限展示用户可访问的故事板列表）→ 选择目标镜头 → 将 output_file_node_id 写入该 Storyboard 的 active_assets JSONB 字段
2. **批量导出：** 批量执行完成后，弹出「导出向导」面板：
   - 自动匹配：如果画布中的 StoryboardNode 记录了 `sourceStoryboardId`，自动建议将对应 GeneratorNode 的产出导出到该镜头
   - 手动调整：用户可以修改、跳过或重新指定每个节点的导出目标
   - 确认后批量执行导出
3. **保存为资产：** 用户右键点击生成结果，选择「保存为资产」→ 创建 AssetResource 记录，将文件关联到项目的资产库中

**导出 vs 原自动挂载的区别：**

| 维度     | 原方案（自动挂载）                    | 新方案（用户主动导出）                     |
| -------- | ------------------------------------- | ------------------------------------------ |
| 触发方式 | 生成完成后自动写入 Storyboard         | 用户确认后手动触发                         |
| 画布独立性 | 画布依赖故事板存在                  | 画布完全独立，导出是可选操作               |
| 灵活性   | 一对一绑定（一个 Generator → 一个镜头） | 同一生成结果可导出到多个镜头或多个项目     |
| 权限     | 隐式（通过画布的 project_id 推导）    | 显式（导出时检查用户对目标项目的写权限）   |

> **注意：** 导出操作需要检查用户对目标项目/故事板的**写权限**（通过 `ProjectMember` 角色），而非仅读权限。画布创作过程中引用数据只需读权限，但导出时需要写权限。

### 5.2 关于「引用组」与右侧资产库的关系

**澄清：**

- 右侧资产库 Tab 是资产的「来源面板」，用户从这里拖拽资产到画布上
- 拖入画布后创建的就是 AssetNode（资产节点）
- AssetNode 的作用是在画布上「代表」一个资产，可以通过连线将其作为 GeneratorNode 的参考输入
- ReferenceNode 的原始设计意图是「引用故事板镜头条目」，但这个功能已经被 StoryboardNode 覆盖

**建议的简化方案：**

- 引用组只保留 AssetNode（资产引用）
- 故事板条目统一使用 StoryboardNode（创作组）
- ReferenceNode 废弃

### 5.4 TextGenNode（文本生成节点）详细设计

#### 5.4.1 设计动机

当前创作工坊的核心痛点之一：**StoryboardNode 中的场景描述是面向人类阅读的自然语言（如「张三站在雨中的街头，远处霓虹灯闪烁」），而 GeneratorNode 需要的是面向 AI 图片/视频模型的结构化提示词（如 `a man standing in rainy street, neon lights in background, cinematic lighting, 8k, detailed`）。** 这两种文本的语义相近但格式差异巨大，目前完全依赖用户手工改写。

TextGenNode 的设计目标是在 StoryboardNode 和 GeneratorNode 之间插入一个 LLM 驱动的「提示词放大器」，自动完成这种转换。同时，该节点也可以独立用于任何需要 LLM 文本生成能力的场景（如角色描述扩写、场景细节补全、多语言翻译等）。

#### 5.4.2 节点定义

| 属性     | 值                                                                              |
| -------- | ------------------------------------------------------------------------------- |
| 节点标识 | `textGenNode`                                                                   |
| 所属组   | AI 生成组                                                                       |
| 输入端口 | `in-text`（text 类型）：接收上游文本（场景描述、剧本片段等）                    |
|          | `in-ref`（asset-ref 类型）：接收资产引用（角色/场景/道具，用于上下文补充）      |
| 输出端口 | `out-text`（text 类型）：输出 LLM 生成的文本，作为下游节点的文本输入            |
| 节点数据 | `systemPrompt`：系统提示词（定义 LLM 角色和输出格式）                           |
|          | `userPromptTemplate`：用户提示词模板（支持变量插值）                            |
|          | `bindingKey`：绑定的 AI 文本模型配置 key                                        |
|          | `presetId`：关联的提示词预设 ID（可选）                                         |
|          | `temperature`：生成温度（默认 0.7）                                             |
|          | `maxTokens`：最大输出 token 数（默认 2048）                                     |
|          | `lastOutput`：最近一次生成的输出文本                                            |
|          | `status`：idle / streaming / succeeded / failed                                 |

#### 5.4.3 HTTP 流式调用机制

TextGenNode 通过前端发起 HTTP SSE 流式请求，调用后端已有的 `/api/ai/text/chat/stream` 接口：

```
前端 TextGenNode 点击「生成」
    ↓
收集上游输入：
  - in-text 端口 → 场景描述文本
  - in-ref 端口 → 资产名称/描述（作为上下文）
    ↓
构造请求体：
  POST /api/ai/text/chat/stream
  {
    "binding_key": "text-default",       // 后端已配置 Key 的文本模型
    "messages": [
      { "role": "system",  "content": systemPrompt },
      { "role": "user",    "content": renderTemplate(userPromptTemplate, context) }
    ],
    "attachments": []
  }
    ↓
前端通过 EventSource / fetch + ReadableStream 消费 SSE 流
    ↓
实时逐 token 渲染到节点内的输出区域（打字机效果）
    ↓
流结束后：
  - 将完整输出写入 node.data.lastOutput
  - 自动传播到下游连线节点的 in-text 端口
  - 状态更新为 succeeded
```

**关键设计决策：**

- **前端交互模式：前端直连后端 SSE，不经过任务系统。** 文本生成通常 2-10 秒完成，无需任务队列的异步调度。这与 GeneratorNode（图片/视频生成耗时 30 秒-5 分钟，必须走任务队列）有本质区别
- **流式渲染。** 用户在节点内实时看到文本逐字生成，体验远优于等待完整响应
- **自动传播。** 生成完成后，输出文本自动沿连线传播到下游 GeneratorNode 的 `in-text` 端口，无需用户手动操作

**双模执行策略（前端 vs 后端批量）：**

TextGenNode 在不同执行场景下采用不同的调用方式：

| 执行场景                           | 调用方式                                             | 流式输出 | 说明                                     |
| ---------------------------------- | ---------------------------------------------------- | -------- | ---------------------------------------- |
| 用户手动点击「生成」               | 前端 SSE 流式请求 `/api/ai/text/chat/stream`         | ✅ 有    | 最佳交互体验，逐 token 打字机渲染        |
| 前端批量执行（BatchQueueManager）  | 前端 SSE 流式请求，**串行**执行                      | ✅ 有    | 同上，但由队列管理器调度                 |
| 后端批量执行（canvas_batch_execute，M3.2） | 后端直接调用 `ai_gateway_service.chat_text()` **非流式** | ❌ 无    | Worker 进程内直接调用，结果写回 CanvasNode |

后端批量执行时，TextGenNode 不走 SSE 流式通道（Worker 进程不需要逐 token 渲染）。后端调用 `chat_text()` 获取完整响应后，将结果写入 CanvasNode.config_json 的 `lastOutput` 字段，并通过 WebSocket 通知前端刷新节点状态。这与 GeneratorNode 的后端执行路径一致（后端调用 AI 网关 → 结果写回 → WS 通知前端）。

#### 5.4.4 提示词模板管理

TextGenNode 的核心能力之一是提示词模板管理，复用后端已有的 `AIPromptPreset` 体系：

**系统预设模板（内置）：**

| 模板名称               | systemPrompt 摘要                                                  | userPromptTemplate 摘要                                        |
| ---------------------- | ------------------------------------------------------------------ | -------------------------------------------------------------- |
| 图片提示词生成         | 你是专业的 AI 绘图提示词专家，将场景描述转化为 Stable Diffusion / DALL-E 风格的英文提示词 | 场景描述：`{scene_text}`，角色：`{asset_names}`，请生成...     |
| 视频提示词生成         | 你是专业的 AI 视频生成提示词专家，关注运动描述和镜头语言           | 场景描述：`{scene_text}`，镜头类型：`{shot_type}`，请生成...   |
| 角色外观描述扩写       | 基于角色名称和简要描述，生成详细的外貌、服装、气质描述             | 角色：`{asset_names}`，已知信息：`{scene_text}`，请扩写...     |

**模板变量自动填充：**

模板中的变量从上游连线节点自动提取：

**用户自定义模板：**

- 用户可以在 TextGenNode 的编辑面板中修改 systemPrompt 和 userPromptTemplate
- 修改后可「保存为模板」，调用 `POST /api/ai/prompt-presets` 写入 AIPromptPreset 表

## 6. 生成节点的参考与提示词机制

### 6.1 当前参考图机制

当前 GeneratorNode 通过端口连线接收参考输入：

- `in-script`（text 类型）：接收上游 StoryboardNode 的场景描述文本
- `in-ref`（asset-ref 类型）：接收上游 AssetNode 的资产引用

但实际的参考图传递存在断层：

- AssetNode 只存储 assetId 和 name，没有实际的图片 URL
- 连线建立的是逻辑关系，但生成请求中的 `images[]` 参数需要实际的图片数据
- 当前 executeTask 回调只传递了 prompt、negPrompt、resolution，没有传递参考图

### 6.2 建议的参考图机制

**参考图来源优先级：**

1. **AssetNode 连线：** 通过 AssetNode 的 assetId 查询后端获取资产的代表图（AssetResource 中 is_default 的图片），转为 data URL 后加入 `images[]`
2. **其他 GeneratorNode 连线：** 如果上游 GeneratorNode 已有生成结果（lastImage），将其作为参考图
3. **用户手动上传：** 在 GeneratorNode 的提示词面板中增加「上传参考图」按钮，用户可以直接上传本地图片
4. **StoryboardNode 的参考图：** 如果 StoryboardNode 有 referenceImageUrl，也可以作为参考

**参考图收集流程：**

- 当用户点击「生成」时，前端遍历所有指向该 GeneratorNode 的 incoming edges
- 按来源类型收集可用的图片 URL
- 将图片下载并转为 data URL（避免跨域问题）
- 打包到生成请求的 `images[]` 参数中
- 限制：最多 4 张参考图，单张不超过 4MB

### 6.3 提示词编辑机制

当前 GeneratorNode 的提示词编辑能力非常有限——只在节点内显示一行 prompt 文本，没有编辑入口。

**建议的提示词编辑方案：**

**方案：节点 Focus 态展开提示词面板**

当用户选中 GeneratorNode 且 zoom ≥ 0.6 时，在节点下方或右侧展开一个提示词编辑面板（屏幕空间浮层，不随缩放变化）：

面板内容：

- 参考来源预览区（横向滚动的缩略图列表，来自连线收集）
- 正向提示词输入框（多行文本，支持提示词模板）
- 负面提示词输入框
- 模型选择下拉框（从 AIModelConfig 拉取可用模型列表）
- 画幅/比例选择（1:1, 16:9, 9:16, 4:3 等）
- 生成模式切换（图片/视频）
- 视频时长选择（仅视频模式）
- 「生成」按钮

这个方案实际上在早期设计文档（方案A）中已经有详细描述（LlmPromptPanel），当前代码中也有 `LlmPromptPanel` 组件的骨架实现，但它只服务于 MediaNode（kind='media'）。需要将其迁移到 GeneratorNode 上。

### 6.4 提示词模板

GeneratorNode 和 TextGenNode 共享提示词模板体系，详见 5.4.4 节。核心要点：

- 系统预设模板（如「角色立绘」「场景全景」「动作特写」等）
- 用户自定义模板（复用现有的 AIPromptPreset 表，通过 `/api/ai/prompt-presets` CRUD）
- 模板变量替换（如 `{scene_text}` `{asset_names}` 自动从连线的 AssetNode/StoryboardNode 中提取，完整变量列表见 5.4.4 节）

### 6.4.1 多输入端口的合并策略

当一个节点的同一输入端口有多条连线时（例如 TextGenNode 的 `in-text` 接收了来自 StoryboardNode 和 ScriptNode 的两条连线），需要明确合并规则：

| 端口数据类型  | 合并策略                                                                 | 说明                                       |
| ------------- | ------------------------------------------------------------------------ | ------------------------------------------ |
| `text`        | 按连线创建时间排序，以 `\n\n---\n\n` 分隔符拼接所有上游文本              | 保留全部上下文，分隔符便于 LLM 区分不同来源 |
| `asset-ref`   | 合并为数组，去重（按 assetId）                                           | 多个 AssetNode 的引用合并为一个资产列表     |
| `image`       | 取**最后连线**的图片（不拼接），但在参考图收集时会收集全部（见 6.2 节）   | 单一主图 + 多参考图的模式                   |

> **注意：** 当前 `propagateData`（`data-flow.ts`）实现中，多条连线写入同一 `targetHandle` 时后写入会覆盖先写入。上述合并策略需要在重构 `propagateData` 时实现——改为按 `targetHandle` 聚合多源输入后再写入。

### 6.5 TextGenNode 与 GeneratorNode 的协同提示词工作流

引入 TextGenNode 后，GeneratorNode 的提示词来源从「用户手写」扩展为三种模式：

**模式 A：手动输入（现有）**

用户在 GeneratorNode 的提示词面板中直接编写 prompt。适合对提示词有精确控制需求的高级用户。

**模式 B：TextGenNode 自动生成（推荐）**

```
StoryboardNode ──(text)──→ TextGenNode ──(text)──→ GeneratorNode
                            LLM 自动转换              自动接收提示词
                            场景描述 → 提示词          无需手动编写
```

TextGenNode 的 `out-text` 输出自动传播到 GeneratorNode 的 `in-text` 端口，覆盖手动输入的 prompt。这是 TextGenNode 的核心使用场景——用户只需在 StoryboardNode 中用自然语言描述场景，LLM 负责将其转化为适合目标模型的提示词格式。

**模式 C：混合模式**

TextGenNode 生成的提示词写入 GeneratorNode 的 prompt 输入框后，用户可以在此基础上手动微调。这种模式兼顾了 AI 效率和人工精确控制：

```
TextGenNode 输出 → 自动填入 GeneratorNode prompt → 用户微调 → 生成
```

**提示词来源优先级（更新）：**

当 GeneratorNode 发起生成时，提示词按以下优先级确定：

1. **用户手动编辑的 prompt**（如果用户在 TextGenNode 输出后又手动修改了）
2. **TextGenNode 输出**（通过 `in-text` 端口传入）
3. **StoryboardNode 场景描述**（如果没有 TextGenNode 中间层，直接连线）
4. **空 prompt**（用户未输入任何提示词，仅依赖参考图）

**手动编辑检测机制：** GeneratorNode 的 `data` 中增加 `promptSource` 字段（`'manual' | 'upstream' | null`）。当 TextGenNode 输出通过 `propagateData` 写入 GeneratorNode 的 prompt 时，`promptSource` 自动设为 `'upstream'`；当用户在提示词面板中手动编辑 prompt 时，`promptSource` 切换为 `'manual'`。生成时根据 `promptSource` 判断优先级，避免无法区分「自动填入」和「手动修改后」的歧义。

**批量执行中的 TextGenNode 调度：**

当用户触发前端批量执行（BatchQueueManager）时，TextGenNode 参与拓扑排序：

```
拓扑排序结果示例：
  Layer 0: [ScriptNode, StoryboardNode(1), StoryboardNode(2), AssetNode(1)]
  Layer 1: [TextGenNode(1), TextGenNode(2)]       ← LLM 文本生成（SSE 流式）
  Layer 2: [GeneratorNode(1), GeneratorNode(2)]    ← 图片/视频生成（任务队列）
```

> **⚠️ 当前 BatchQueueManager 需要重构：** 现有实现（`batch-queue.ts`）硬编码过滤 `generatorNode`，会丢弃其他类型节点。需改为支持多种可执行节点类型，并根据节点类型选择不同的执行策略：

```typescript
// 重构方向（伪代码）：
const EXECUTABLE_TYPES = {
  textGenNode:   { strategy: 'sse-stream', maxConcurrency: 1 },
  generatorNode: { strategy: 'task-queue', maxConcurrency: 3 },
};
// enqueue 时按 EXECUTABLE_TYPES 过滤，而非硬编码 generatorNode
```

TextGenNode 在批量模式下的特殊处理：

- **串行执行 LLM 调用**（`maxConcurrency: 1`），避免同时发起多个 SSE 连接导致前端资源压力
- **每个 TextGenNode 完成后立即传播**输出到下游，不等待同层其他 TextGenNode
- **超时控制**：单个 TextGenNode 超时 30 秒（LLM 文本生成通常 2-10 秒）
- **失败回退规则**：TextGenNode 失败时，下游 GeneratorNode 沿 `in-text` 连线链**向上回溯到最近的非 TextGenNode 上游节点**，使用该节点的输出文本作为 fallback。典型场景：StoryboardNode → TextGenNode(失败) → GeneratorNode，回退到 StoryboardNode 的 `out-text` 输出。如果上游也是 TextGenNode（链式串联场景），则继续回溯直到找到非 TextGenNode 源头

## 7. 任务系统并发与多人协作能力评估

### 7.1 当前并发能力

**后端 Worker：**

- 单进程异步循环，通过 `brpop` 从 Redis 队列逐个取任务
- 每个任务在 asyncio 事件循环中执行，理论上支持 I/O 并发（等待 AI API 响应时不阻塞其他任务）
- 但实际上 `_worker_loop` 是串行取任务的：取一个 → 处理完 → 取下一个
- 任务超时控制：图片生成 5 分钟，视频生成 30 分钟，其他 10 分钟

**前端 BatchQueueManager：**

- maxConcurrency 默认 3，即同时最多 3 个节点在执行
- 拓扑排序保证依赖关系
- 超时检测（默认 300 秒）

**实际并发瓶颈：**

| 层级         | 并发能力                          | 瓶颈               |
| ------------ | --------------------------------- | ------------------ |
| 前端队列     | 3 个并发请求                      | 可配置，不是瓶颈   |
| Next.js BFF  | 无限制（受 Node.js 事件循环限制） | 不是瓶颈           |
| FastAPI 后端 | 异步框架，理论上高并发            | 不是瓶颈           |
| Redis 队列   | 单队列，brpop 串行消费            | **主要瓶颈** |
| Worker 进程  | 单进程，串行取任务                | **主要瓶颈** |
| AI API 调用  | 受厂商 QPS 限制                   | 外部瓶颈           |
| TextGenNode SSE | 前端串行（maxConcurrency: 1）   | 前端瓶颈（新增）   |

**结论：** 当前架构下，后端 Worker 是串行处理任务的。虽然单个任务内部是异步的（等待 AI API 时不阻塞 CPU），但 Worker 不会同时处理多个任务。这意味着：

- 如果有 10 个图片生成任务排队，它们会一个接一个执行
- 前端的 maxConcurrency=3 只是控制「同时发起请求」的数量，但后端仍然串行处理
- 实际吞吐量取决于单个 AI API 调用的耗时

**TextGenNode 引入的额外并发考量：**

- **前端 SSE 资源限制：** 浏览器对同域名的并发连接数通常限制为 6 个（HTTP/1.1）或更多（HTTP/2）。TextGenNode 批量串行执行（maxConcurrency: 1）避免了此问题，但如果未来放宽并发，需注意 SSE 连接占用
- **LLM API Rate Limit：** 文本模型 API 通常有 TPM（tokens per minute）限制。批量执行 20 个 TextGenNode（每个约 2000 tokens 输出）将消耗约 40K tokens，可能触发厂商限流。建议在 TextGenNode 批量执行间增加可配置的间隔（默认 500ms）
- **Token 用量追踪（M3.4）：** 每次 TextGenNode 执行后，将 LLM 返回的 `usage.total_tokens` 记录到 CanvasExecution.result_summary 中，便于用户了解成本。前端在批量执行完成后的汇总面板中展示总 token 消耗

### 7.2 多人同时工作的能力

**当前不支持多人协作。** 具体表现：

- 画布数据存储在 VFS JSON 文件中，没有并发控制
- 如果两个用户同时编辑同一个画布，后保存的会覆盖先保存的（Last Write Wins）
- 没有实时同步机制（如 CRDT 或 OT）
- 没有画布级别的锁定机制

**多人场景下的风险：**

- 数据丢失：两人同时编辑，一人的修改被覆盖
- 任务冲突：两人同时对同一个节点发起生成，产生重复任务
- 状态不一致：一人看到的节点状态与另一人不同

### 7.3 提升并发能力的建议

**短期（Worker 多实例）：**

- 启动多个 Worker 进程，每个进程独立从 Redis 队列取任务
- Redis `brpop` 天然支持多消费者竞争消费，无需修改队列逻辑
- 建议启动 3-5 个 Worker 进程，与前端 maxConcurrency 匹配
- 实现方式：在 docker-compose 中将 worker 服务的 `replicas` 设为 3-5

**中期（任务优先级队列）：**

- 引入多个 Redis 队列，按优先级区分（如 high/normal/low）
- 图片生成走 high 队列（用户等待中），批量任务走 normal 队列
- Worker 优先消费 high 队列

**长期（多人协作）：**

- 画布数据从 VFS JSON 迁移到数据库（Canvas + CanvasNode 模型）
- 引入 WebSocket 实时同步（基于操作日志的 OT 或 CRDT）
- 画布级别的乐观锁或悲观锁
- 这是一个重大架构变更，建议作为独立里程碑规划

## 8. 导出与剪辑工作流设计

### 8.1 导出需求分析

创作工坊的最终产出是一系列按镜头组织的媒体文件（图片、视频、音频），用户需要将这些产出导出到专业剪辑软件（如 Premiere Pro、DaVinci Resolve、Final Cut Pro）中进行后期制作。

导出的核心挑战：

- 如何保持镜头的顺序和结构信息
- 如何让剪辑软件能够直接识别和使用这些素材
- 如何减少用户在剪辑软件中的重复性操作

### 8.2 建议的导出方案

**层级一：素材包导出（基础）**

将画布中所有生成结果按镜头顺序打包为 ZIP 文件：

```
导出包/
  ├── manifest.json          # 元数据（镜头列表、时长、资产关联）
  ├── SC01/
  │     ├── SH01_image.png   # 镜头 1 的图片
  │     ├── SH01_video.mp4   # 镜头 1 的视频
  │     └── SH01_audio.wav   # 镜头 1 的音频（如有）
  ├── SC02/
  │     ├── SH02_image.png
  │     └── SH02_video.mp4
  └── assets/
        ├── 张三_角色设计.png  # 使用到的资产
        └── 李四_角色设计.png
```

**层级二：时间线 XML 导出（进阶）**

生成 FCP XML（Final Cut Pro XML Interchange Format）或 AAF 格式的时间线文件，剪辑软件可以直接导入：

- 按镜头顺序排列视频片段
- 自动设置每个片段的入点和出点
- 保留镜头编号和场景描述作为标记（Marker）
- 音频轨道独立（如有配音）

FCP XML 是目前兼容性最好的跨软件时间线格式，Premiere Pro、DaVinci Resolve、Final Cut Pro 都支持导入。

**层级三：在线预览与标注（远期）**

在平台内提供简易的时间线预览功能：

- 按镜头顺序播放所有视频片段
- 支持添加文字标注和修改意见
- 支持导出带标注的审片报告
- 这个功能可以减少「导出 → 剪辑软件 → 审片 → 反馈」的循环次数

### 8.3 导出流程设计

```
用户点击「导出」
    ↓
选择导出格式（素材包 / FCP XML / 在线预览）
    ↓
选择导出范围（全部镜头 / 选中镜头 / 指定场次）
    ↓
后端创建导出任务（新增 canvas_export 任务类型）
    ↓
Worker 执行：
  1. 从 VFS/MinIO 收集所有相关媒体文件
  2. 按格式要求组织文件结构
  3. 生成 manifest.json 或 FCP XML
  4. 打包为 ZIP 并上传到 MinIO exports bucket
    ↓
返回下载链接，用户下载
```

### 8.4 与现有系统的集成

- 导出任务复用现有 Task 系统，新增 `canvas_export` 任务类型和对应 Handler
- 导出文件存储在 MinIO 的 `anyreason-exports` bucket（已存在）
- 导出记录可以关联到 Canvas 和 CanvasExecution

## 总结

创作工坊当前面临的核心问题不是技术实现的缺陷，而是产品定位和架构设计的模糊。通过明确**「画布是系统中独立的、自治的创作单元」**这一核心定位，可以推导出一系列清晰的设计决策：

- **画布独立化**：画布不绑定任何项目或分集，是用户级别的独立资源。创作者在创意工坊全屏界面中对画布进行增删改查，创作过程中根据权限引用剧本、故事板和资产作为参考
- **两级全页界面**：画布列表页（`/studio`，卡片式 CRUD）+ 画布编辑页（`/studio/[canvasId]`，全屏沉浸），右侧参考数据浏览器按权限拉取多项目的剧本/故事板/资产
- 精简节点体系（10 → 8），合并 3 个冗余节点，**新增 TextGenNode（文本生成节点）**——通过 LLM 流式调用自动将场景描述转化为高质量媒体生成提示词，支持提示词模板管理和双模执行策略（前端 SSE / 后端非流式）
- **统一端口命名规范**（`in-{type}` / `out-{type}`），修正现有代码中端口命名不一致的问题，并为 TextNoteNode、CandidateNode 补齐缺失的输出端口
- **VFS JSON 为权威数据源、数据库为查询索引**的双写一致性策略，明确分离运行时数据（lastOutput）与静态配置（config_json）
- **数据流引擎增强**：支持多源输入合并（text 拼接、asset-ref 去重合并）、`promptSource` 手动编辑追踪、TextGenNode 失败时的连线链回溯 fallback
- **弱引用数据模型**：Canvas 表无 project_id/episode_id 外键，CanvasNode 的 source_storyboard_id/source_asset_id 无 FK 约束，节点内保存快照数据，不依赖源数据持续存在
- **用户主动导出**替代自动挂载：画布产出独立存储，仅在用户主动「导出到故事板」或「保存为资产」时建立关联，导出时检查目标项目的写权限
- 后端化的任务调度，因为批量生成和定时执行是创作流程的刚需
- 标准化的导出格式，因为故事板的最终归宿是专业剪辑软件

建议按 M1 → M2 → M3 → M4 → M5 的顺序推进，每个里程碑都是可独立交付的增量。工作计划经覆盖度审查（见 9.1 节），已修正 6 项遗漏和 3 项冲突：

- **M1**（基础重构，5 个子任务）：明确了子任务执行顺序（M1.1/M1.5 并行 → M1.2 → M1.3 → M1.4），M1.1 画布列表在 M2 就绪前使用 VFS 驱动过渡，M1.2 补充了存量画布数据迁移，M1.4 的 BatchQueueManager 重构采用接口抽象避免与 M3.2 冲突
- **M2**（数据模型与后端，5 个子任务）：新增 M2.4 参考数据权限 API 和 M2.5 VFS/DB 双写同步实现
- **M3**（任务系统增强，4 个子任务）：新增 M3.4 Token 用量追踪与启动校验
- **M4**（文件归属与导出，4 个子任务）：新增 M4.4 画布缩略图生成
- **M5**（导出能力）：M5.1 素材包导出兼容无故事板引用的独立画布
