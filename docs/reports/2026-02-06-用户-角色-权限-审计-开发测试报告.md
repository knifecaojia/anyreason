## 1. 目标与交付

本次交付实现：

1) 用户管理：用户列表展示、为用户分配角色（支持 0..N 角色；前端当前以“主角色下拉”方式写入 0/1 角色）。

### 2.2 权限控制策略

鉴权：沿用 `fastapi-users` 的 JWT 登录与 `current_active_user`。  
授权：在业务层增加 RBAC 校验，规则如下：

- `is_superuser=True` 的用户天然拥有所有权限（用于系统初始管理与紧急兜底）。
- 非 superuser 用户：通过 `user_roles -> role_permissions -> permissions.code` 计算权限集合。
- 关键管理接口使用 `require_permissions([...])` 做门禁。

### 2.3 管理 API（/api/v1/admin）

路由文件：

- [admin.py](file:///F:/animate-serial/apps/anyreason/fastapi_backend/app/api/v1/admin.py)

接口概览（部分）：

- `GET /api/v1/admin/users`：用户列表（含角色）  
- `PUT /api/v1/admin/users/{user_id}/roles`：设置用户角色（body: role_id 数组）  
- `GET /api/v1/admin/roles`：角色列表（含权限）  
- `POST /api/v1/admin/roles` / `PATCH` / `DELETE`：角色维护  
- `GET /api/v1/admin/permissions`：权限字典  
- `POST /api/v1/admin/permissions` / `PATCH` / `DELETE`：权限维护  
- `PUT /api/v1/admin/roles/{role_id}/permissions`：设置角色权限（body: permission_id 数组）  
- `GET /api/v1/admin/audit-logs`：审计日志列表（limit/offset）  
- `GET /api/v1/admin/audit-logs/count`：审计日志总数

审计覆盖：

- 角色/权限 CRUD、角色授权、用户分配角色均写入 `audit_logs`（含 request_id、ip、path 等上下文）。

## 5. 已知限制与后续建议

1) 用户管理范围：目前实现了“列用户 + 分配角色”。如需“禁用账号/重置密码/创建用户邀请”，建议复用 `fastapi-users` 的 `PATCH /users/{id}` 与邮件能力，并在前端补齐对应交互。  
2) 审计覆盖范围：当前聚焦“管理操作审计”。如需“全量接口审计/行为审计”，建议再引入中间件级别的记录策略，并明确敏感字段脱敏与合规保留期。  
3) 权限粒度：当前采用 `permissions.code`（字符串）作为权限原语，推荐以 `system.* / project.* / asset.*` 分层扩展，并建立“权限字典基线种子数据”（初始化脚本）减少人工创建成本。
