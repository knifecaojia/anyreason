# Spec06（资产三层存储 / 变体文档 / 重分集安全化）实施方案与计划

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** 以“资产数据”为核心，把 AI 提取的 VFS 文档与 DB 资产/变体/资源三层结构打通；同时把“重分集”升级为默认非破坏性（不删资产、不删故事板），必要时仅清理/重建“绑定关系”，并提供可回滚与可审计能力。

**Architecture:** 现状已经具备三层资产表（assets / asset_variants / asset_resources）与 VFS（file_nodes）。本方案补齐“变体=唯一文档”绑定能力与 AI 落库链路的幂等/关联；并把资产与故事板视为“剧本/项目级资源”，把剧集关联下沉到“绑定关系”（asset_bindings、storyboard.episode_id）这一层：重分集时默认不做强制覆盖，仅更新/新增剧集并按策略处理绑定，从而避免误伤。

**Tech Stack:** FastAPI + SQLAlchemy(Async) + Alembic + Postgres(JSONB/partial index) + Next.js(BFF routes) + VFS(file_nodes + MinIO)

## 0. 现状结论（基于代码核对）

### 0.1 三类核心实体与关系（已存在）
- **剧本 Script（scripts）**：已软删除（is_deleted/deleted_at），以 MinIO 作为原文 SSOT，并衍生 Project(id=script_id) 作为工作空间根。
- **剧集 Episode（episodes）**：保存 script_full_text（分集后的正文段）、episode_doc_node_id（分集 markdown 文件指针）、asset_root_node_id / storyboard_root_node_id（产物目录根指针）。
- **资产三层结构（已基本满足 spec06 “三层存储”）**
  - **Asset（assets）**：资产实体（角色/道具/地点/特效），并已支持 doc_node_id（资产说明书 markdown 指针）。
  - **AssetVariant（asset_variants）**：变体/版本承载，但当前不支持 doc_node_id（缺口）。
  - **AssetResource（asset_resources）**：资源文件（图片等）记录，meta_data 里当前放了 file_node_id 与 is_cover 等（可用但需规范化/约束）。

### 0.2 VFS 与业务表的绑定方式（已存在）
- 业务表保存 `file_nodes.id` 作为“指针字段”，内容本体在 MinIO；前端通过 `/vfs/nodes/{id}/download` 打开 markdown。
- ApplyPlan 已实现：
  - `episode_save`: 写 `分集/*.md` 并回写 `episodes.episode_doc_node_id`
  - `asset_create`: 写 `资产/<类型>/*.md` 并创建/补齐 DB Asset，同时写/补齐变体（但变体没有 doc_node_id）
  - `asset_bind`: 仅写 `绑定/*.json`，不落 DB 绑定（缺口）

### 0.3 重分集（force=1）当前是破坏性覆盖（需要改）
- 正则结构化会删除不再存在的 Episode，并对每个 Episode：先删 Storyboard 再重建。
- 由于 Storyboard 下面级联 prompts / bindings 等，任何重分集都会直接“误伤”已完成的资产提取与故事板创作。

## 1. 需求拆解与目标边界（spec06 对齐）

### 1.1 spec06 关键需求点
1) 对剧本/剧集/资产/故事板 DB 存储做详细梳理（本文件第 0 节 + 附录）。
2) 自动分集可多次点击：必须定义“重复执行如何影响数据”，并在系统层面避免误伤。
3) AI 助手提取的 VFS 文件应同时创建资产数据：以资产数据为核心展示剧本/剧集内容。
4) 资产存储三层：资产（Asset）/资产变体（Variant）/资产实体文件（Entity File=图片等）。
5) 变体绑定规则：每个资产变体表有且仅有一个 markdown vfs，用于描述该变体。
6) 图片筛选：勾选进入“资产实体表”；star 作为首图，且排他。

### 1.2 非目标（本阶段不做）
- 不引入“完整不可变版本史/审计系统”（可通过 VFS 备份实现轻量回滚）。
- 不强制重构现有前端为“完全资产中心化 UI”，仅补齐“可引用/可跳转/可绑定”的数据基础与接口。

## 2. 方案对比（通过 superpowers 方法论：现状证据 → 方案对比 → 风险收敛）

### 2.1 重分集策略（核心分歧点）

**方案 A：默认非破坏性重分集（不删除资产/故事板），仅更新剧集；绑定可选清理/保留**（推荐）
- 核心语义：重分集改变的是“剧本文本 → 剧集切分结果”，不是资产/故事板本体。资产与故事板可视为项目级公共资源；当切分变化导致“旧绑定不再可信”时，只处理绑定关系。
- 优点：彻底消除“误删作品”的风险；贴合你的期望——重分集后资产/故事板依然存在，只是绑定关系可能需要重新指定（可在资产卡片中重新绑定到新剧集）。
- 缺点：需要补齐“重新绑定”的产品路径（至少 Episode 维度的 AssetBinding upsert 与 Storyboard 重新归属能力）。

**方案 B：引入“结构版本（structure_run）”，让旧版本保留，UI 默认展示最新**（中期可选）
- 优点：真正的安全：重分集不会删旧数据；可在 UI 切换结构版本；天然可回溯。
- 缺点：需要 DB 扩展（Episode/Storyboard 与 run_id 关联、唯一约束调整）、前端展示逻辑调整，范围更大。

**方案 C：智能合并（只更新未被人工编辑的 storyboard/绑定）**
- 优点：用户体验最好。
- 缺点：需要新增“生成来源/编辑标记/变更检测”，否则无法可靠判断哪些可覆盖；难度与不确定性最大。

**推荐路径：A（默认） → B（可选）**
- 先落 A：把重分集的“破坏性覆盖”改成“绑定层重建”，风险最低且符合“资产/故事板公共化”的导演工作方式。
- 再评估 B：当你需要“随时查看旧切分版本”或“并行多个切分方案”时再升级。

### 2.2 变体文档绑定（实现策略）

**方案 1：给 asset_variants 增加 doc_node_id（最贴合 spec06）**（推荐）
- 每个 Variant 绑定 1 个 markdown VFS 文件：`asset_variants.doc_node_id -> file_nodes.id`
- 可加 DB 约束：对同一 variant，doc_node_id 要么为空要么唯一。

**方案 2：不改 DB，只在 VFS 用命名规则隐式映射**
- 优点：无迁移。
- 缺点：无法强一致（rename/move 破坏映射）；难以在 API/前端稳定引用；不满足“以资产数据为核心”的落库要求。

推荐：方案 1。

## 3. 具体落地设计（数据库 / VFS / API / 任务链路）

### 3.1 数据库改造（最小可行）

#### 3.1.1 asset_variants 增加 doc_node_id
- 新增列：`asset_variants.doc_node_id UUID NULL`
- 外键：`FK asset_variants.doc_node_id -> file_nodes.id ON DELETE SET NULL`
- 索引：`idx_asset_variants_doc_node_id`
- 业务约束（应用层）：一个 variant 只允许绑定一个 doc_node_id（天然满足）；同一 variant 的 doc_node_id 只能指向“文件节点（非 folder）”。

#### 3.1.2 asset_resources 的“首图 star 排他”规范化
当前 `is_cover` 放在 `meta_data`，且写入时不清理旧 cover，可能出现多 cover。

建议（两步走）：
1) 增加列：`asset_resources.is_cover boolean NOT NULL default false`
2) 增加约束：`UNIQUE (variant_id) WHERE is_cover = true`（partial unique index）
3) 回填迁移：从 `meta_data->is_cover` 回填到列，并在写入时只写列；`meta_data.is_cover` 逐步废弃。

### 3.2 VFS 布局（兼容存量）

保持现有资产文档不变（asset 级）：
- `资产/角色/character_张三.md` 继续作为“资产总览/摘要”文档（现存 UI 依赖 `assets.doc_node_id`）。

新增变体文档（variant 级）：
- 建议路径：`资产/<类型中文>/<资产名>/variants/<variant_key>.md`
  - `<资产名>`：safe_filename(asset.name)
  - `<variant_key>`：优先 stage_tag（如 “战损版”），否则 variant_code（如 V2）
  - 写入时同时更新 `asset_variants.doc_node_id`

### 3.3 ApplyPlan/AI 落库链路增强

#### 3.3.1 asset_create：从“资产级 doc”扩展到“变体级 doc”
- 当前：对 AssetDocV2 渲染并写 `assets.doc_node_id`，并在 DB 创建/补齐 variants（但无 doc_node_id）。
- 改造：对 `AssetDocV2.variants[]`：
  - 为每个变体生成 variant markdown（新增 renderer）
  - 通过“定位/创建 variant”的逻辑拿到 `asset_variants.id`，并写入 `asset_variants.doc_node_id`
  - 若输入只给了 `stage_tag`：以 `(asset_entity_id, stage_tag)` 优先匹配；否则以 `(asset_entity_id, variant_code)`

#### 3.3.2 asset_bind：从“只写 JSON”升级为“写 JSON + 落 DB 绑定”
- JSON 继续写入 `绑定/EPxxx_bindings.json`（作为审计与可回放输入）。
- 同时落 DB：
  - Episode 维度：写 `asset_bindings (episode_id, asset_entity_id, asset_variant_id?)`
  - Storyboard 维度（如果 bindings_doc 扩展到 shot_code/storyboard_id）：写 `asset_bindings (storyboard_id, asset_entity_id, asset_variant_id?)`

### 3.4 重分集安全化（按方案 A 的“公共资源 + 绑定重建”语义落地）

新增结构化接口语义（保持兼容，且默认不再强制覆盖）：
- `POST /api/v1/scripts/{script_id}/structure?mode=...&binding_policy=...&storyboard_policy=...`
  - `mode=preview`：只返回解析结果，不写 DB
  - `mode=apply_non_destructive`（默认）：
    - **不删除 Episode**：对不再存在于解析结果的 episode_code，标记 `status='orphaned'`（或 stage_tag），用于 UI 隐藏/筛选
    - **不删除 Storyboard**：默认不动 storyboards；可选 `storyboard_policy=reseed_if_empty`（仅当该 episode 没有任何 storyboard 时，才按解析结果补种）
    - **绑定策略（核心）**：由 `binding_policy` 决定是否清理/保留
      - `preserve`：保持现有 asset_bindings 不变（最保守）
      - `clear_episode_bindings`：清理 episode 维度绑定（保留 storyboard 维度绑定）
      - `clear_all_bindings`：清理 episode+storyboard 两类绑定，让用户后续在资产/故事板卡片重新指定
  - `mode=apply_force`（保留但不推荐默认使用）：兼容旧行为（删 Episode/删 Storyboard 重建），但执行前必须写 VFS 备份 + 互斥锁

备份输出（VFS）建议：
- `备份/structure_runs/<timestamp>_<script_id>/episodes.json`
- `备份/structure_runs/<timestamp>_<script_id>/storyboards.json`
- `备份/structure_runs/<timestamp>_<script_id>/bindings.json`

## 4. 影响分析

### 4.1 后端影响面
- DB：需要新增 1-2 个迁移（asset_variants.doc_node_id；asset_resources.is_cover + partial unique）。
- API：apply_plans 行为变化（asset_create / asset_bind），scripts 结构化接口新增 mode/preview/safe 语义。
- Service：asset_service.create_resources 需要保证 cover 排他（写 is_cover 列 + 清理旧 cover）。
- VFS：新增 variant docs 的写入目录与备份目录，file_nodes 数量会上升。

### 4.2 前端影响面
- assets 页面：建议补齐“绑定到剧集”的入口（资产卡片里选择 Episode，并选择绑定到资产/默认变体或指定变体），用于重分集后的重新绑定。
- storyboards 展示：需要一个“未归属故事板/公共故事板”视图，或提供把 storyboard 重新归属到 episode 的交互。
- scripts 页面：可逐步把 `@资产名`、`资产调用` 等从纯文本升级为“可点击跳转到资产/变体文档”。
- 重分集按钮：默认走 `apply_non_destructive + clear_episode_bindings`（或由你决定默认策略），并保留高危 force 的二次确认入口。

### 4.3 数据兼容性
- 存量 assets.doc_node_id 继续可用。
- 新增 asset_variants.doc_node_id 默认 NULL，不会破坏现有接口。
- cover 从 meta_data 迁移到列，需要兼容读取：短期允许两者并存，优先读列。

## 5. 风险分析与缓解

### 5.1 破坏性覆盖导致数据丢失（最高风险）
- 缓解：引入 mode；默认 safe；force 前写 VFS 备份；前端二次确认；后端互斥锁。

### 5.2 并发重入导致结构化结果“最后写入 wins”
- 缓解：script_id 维度互斥锁（DB advisory lock 或应用层锁）；并把本次 run_id 写入备份路径以便追踪。

### 5.3 变体文档命名冲突/重命名导致引用漂移
- 缓解：DB 以 doc_node_id 为准（引用稳定）；文件名可变更但不影响绑定。

### 5.4 cover 排他约束引发写入失败（中风险）
- 缓解：服务层先把同 variant 的旧 cover 清零，再设置新 cover；并提供“显式设置封面”接口用于修复脏数据。

### 5.5 回填迁移耗时/锁表
- 缓解：迁移只做轻量字段与索引；回填用分批脚本/后台任务执行（不在 DDL 里扫全表）。
